# Engine contracts — the whole interface

This is the only engine file you normally need to read. The engine is a black
box; you build a game by writing `game/` (glue), not by editing `engine/`.

## The verbs (import from engine/core)
- `signal(v)` → `{ get(), set(v), update(fn), subscribe(fn) }` — one reactive value.
- `makeRegistry()` → `{ add(e), remove(e), each(fn), query(pred), has(e), size, flush(), clear() }` — a set of entities; add/remove are deferred (safe to mutate while iterating).
- `on(name, fn)` / `off(name, fn)` / `emit(name, data)` — the event bus. `on` returns an unsubscribe fn.
- `spawn(kind, opts)` / `spawnInto(reg, kind, opts)` / `despawn(e)` — entities. `despawn` removes from the scene AND the entity's registry.
- `makeRng(seed)` → `{ next(), range(a,b), int(a,b) }` — deterministic RNG.
- `makeLoop({ paused, registries })` → `{ start(), stop(), step(dt) }` — the frame driver.

## Entity fields
`{ kind, pos:[x,y,z], rot, scale (number or [x,y,z]), vel:[x,y,z], color:[r,g,b] (0..1), mesh, radius, dead, ...your fields }`.
`mesh` is one of: `box | sphere | cylinder | plane`. `spawn(opts.at)` sets pos.

## Phases (emitted by the loop each frame, in order)
`input → update(dt) → physics → late` (the sim; skipped when paused) then `render(dt)` (always).
Continuous state → **signals** (input sticks). Discrete happenings → **events**.

## Signals the systems expect (define these in game/contracts.js)
`gameState` (menu|playing|paused|over) · `score` · `wave` · `playerHealth` · `moveInput` [x,z] · `aimInput` [x,z] · `firing` · `player` (the entity).

## Registries the systems expect
`enemies` · `bullets`.

## Events (the vocabulary)
Raised by systems: `game-started` · `game-over` · `player-fired` · `enemy-hit {enemy,bullet}` · `enemy-died {enemy}` · `player-hit {enemy}` · `player-damaged {hp}` · `player-died` · `wave-started {n}` · `wave-cleared {n}`.

## Systems (engine/systems/*, each imports only core)
`input` (touch→sticks) · `movement` (input+velocity→position) · `ai` (runs each enemy/NPC's behavior) · `fire` (spawn bullets) · `spawn` (wave director) · `collision` (overlap→events) · `health` (hits→damage/death) · `score` · `cleanup` (life/bounds despawn) · `gamestate` (lifecycle+pause) · `hud` (DOM overlay) · `audio` (WebAudio SFX) · `render` (WebGL follow-cam).

## Cameras (reusable component — engine/render/cameras.js)
Set `ctx.camera` in bootstrap to pick a view; all are swappable without touching game logic:
`cameras.topDown({height,back})` · `cameras.sideScroller({distance,height,ahead})` · `cameras.thirdPerson({distance,height})` · `cameras.orbit({radius,height,speed})` · `cameras.fixed({eye,target})` · `cameras.flat2D({height,size})` (true 2D, orthographic top-down) · `cameras.sideScroller2D({distance,size})` (2D platformer). Each is a pure `(focus, entity, dt) → {eye,target,up?,projection?,orthoSize?}`; add your own the same way.

## Movement (reusable component — engine/control/movements.js)
Set `ctx.movement` in bootstrap: `movements.twinStick({speed})` (top-down) · `movements.eightWay({speed})` · `movements.tank({speed,turn})` · `movements.platformer({speed,jump,gravity})` (side view, gravity+jump) · `movements.autoRun({speed,jump})` (endless runner). Each is `update(entity, input, dt)` where `input = {move:[x,z], aim:[x,z], jump}`. Default is twinStick.

## Weapons (reusable component — engine/control/weapons.js)
Set `ctx.weapon` in bootstrap to pick the gun (rhythm + pattern): `weapons.single({cooldown})` (default) · `weapons.rapid({cooldown})` (machine gun) · `weapons.shotgun({pellets,spreadDeg,cooldown})` (fan) · `weapons.burst({count,gap,cooldown})` (tap-fire) · `weapons.radial({ways,cooldown})` (nova/bomb). Each is a stateful `update(dt, firing, dir) → shots[]`; a shot is `{dir:[x,z], speed?, life?, damage?, scale?}`. The fire system spawns whatever it returns, so this is the one place that decides "what a shot is." Per-shot `damage` overrides `config.bulletDamage`.

## Aim (reusable component — engine/control/aim.js)
Set `ctx.aim` in bootstrap to pick how shots are pointed: `aim.stick()` (twin-stick, default) · `aim.facing()` (shoot where you move — one-stick/auto-fire) · `aim.manual()` (sticky: flick to set, holds after release) · `aim.autoAim({range})` (lock the nearest enemy — mobile aim assist). Each is `resolve(player, raw, ctx) → [x,z]` (a unit direction). The fire system asks the aimer each shot.

## Behaviors (reusable component — engine/control/behaviors.js)
The brain for each enemy/NPC (in the `enemies` registry). Set a game-wide default `ctx.behavior`, or give one entity its own `e.behavior`, or mix types with `behaviors.byKind({kind: ..., default: ...})`. Enemy: `chase()` (default) · `flee()` · `orbit({radius,dir})` (circle-strafe) · `keepDistance({min,max})` (ranged) · `zigzag({amp,freq})` (weaves) · `charger({range,windup,dashSpeed})` (telegraph→dash). NPC: `wander({area})` (roam near spawn) · `patrol({points})` (waypoint route) · `follow({distance})` (companion) · `guard({radius})` (hold home, chase intruders). Each is `step(entity, target, dt, ctx)` and sets velocity + facing; per-entity state lives on `e._ai`, so one instance drives a whole registry. `target` is the player, or null when it's gone (NPC behaviors keep going; chasers idle).

## Spawn directors (reusable component — engine/control/spawners.js)
The "when and what to spawn" brain, swappable like cameras/movements/behaviors. Set `ctx.director` in bootstrap; the spawn system calls it each update phase as `director(dt, api)`. A director is a PURE stepper — it never imports world/rng, it only drives the injected `api = { count() (live enemy count), spawn(opts), setWave(n), rng, onWave(cb) }`, so it unit-tests with a recording fake. Variants: `waves({base,growth,delay})` (escalating arena rounds; advances to the next, bigger wave once `count()===0` and a `delay` breather passes — batch n = base+growth*(n-1)) · `endless({rate,max})` (steady time-based trickle, catches up long frames, `max` caps live count) · `burst({size,every,max})` (periodic clumps with lulls) · `boss({atWave,make,escort})` (watches the wave via `onWave`, spawns one boss + escorts once, then steps aside) · `timed({schedule:[{at,count,make}],loop})` (scripted absolute-time cues for intros/boss-rush). Compose with `chain(...directors)` — it forwards any wave announced via `setWave` into every child's `onWave`, so `chain(waves(), boss({atWave:5, make}))` lets a boss watch a wave loop. `spawn(opts)` shape is the game's own contract (opts carries { wave } | { pack } | { boss, make, escort } | { make, at }).

## Pickups (reusable component — engine/control/pickups.js)
What an item does when the player collects it — a swappable effect component. Set it per-drop on an item entity (`item.pickup = pickups.health({ amount: 50 })`), not game-wide. Each factory returns `{ kind, apply(ctx, player) }`; `apply` mutates the game's signals and/or stashes a timed buff on the player, then returns a short tag string ('heal'|'shield'|'score'|'speed'|'weapon'|'magnet'|'life') for a label/sound. Variants: `health({amount,cap})` (raise playerHealth, capped) · `shield({duration})` (player.shield invuln timer) · `scoreBonus({points})` · `speedBoost({mult,duration})` (player.speedMult+speedTimer for movement) · `weaponSwap({weapon})` (set ctx.weapon) · `magnet({radius,duration})` (player.magnetRadius+magnetTimer) · `extraLife({refill,cap})` (bump lives signal/player.lives, optional heal). Helper `makePickup(kind, apply)`. Timed buffs are seconds-remaining fields a tiny game system counts down and clears.

## Hazards (reusable component — engine/control/hazards.js)
Environmental effect zones — lava, spikes, wind, pits — swappable like behaviors. A game places a zone in the world as either a circle `{pos:[x,y,z], radius}` or an axis-aligned box `{min:[x,z], max:[x,z]}`, then picks a hazard per zone: `const lake = hazards.lava()`. Each hazard exposes `inside(entity, zone) -> bool` (radius-aware membership, works for both shapes) and `affect(entity, dt, zone) -> {damage?, push?[x,z], kill?, slow?} | null`. The verdict is PURE DATA — the hazard never mutates the entity; a system reads the fields and applies them (subtract `damage`, add `push` to vel, despawn on `kill`, scale speed by `slow` 0..1). Returns null when the entity is outside. Variants: `damageZone({dps=10})` (dt-scaled DoT) · `spikes({damage=20})` (one-shot flat hit on entry, re-arms after you leave — per-entity via WeakSet) · `lava({dps=30,slow=0.5})` (DoT + drag) · `slowField({mult=0.5})` (pure speed multiplier — mud/web) · `pit({})` (instant kill inside) · `windZone({force:[x,z]})` (constant dt-scaled velocity push — conveyor/fan/current; also exposes `outward(entity,zone)` for radial repulsors). Pure and sim-testable; a game applies verdicts wherever it steps hazards (e.g. a small system in `update`).

## Difficulty (reusable component — engine/control/difficulty.js)
Set `ctx.difficulty` in bootstrap to pick how the game ramps up. Each is a pure `scale(state) -> {speedMul, hpMul, rateMul, damageMul}` where `state = {time, score, wave, performance}`; the spawn director multiplies enemy stats by the muls when building a wave. `difficulty.flat({speed,hp,rate,damage})` (fixed, a casual baseline) · `difficulty.linear({per,cap,by,unit})` (smooth continuous ramp vs time/wave/score, default) · `difficulty.stepped({every,step})` (discrete plateaus that jump) · `difficulty.waveBased({perWave})` (keyed to the wave counter; wave 1 = base) · `difficulty.adaptive({up,down,floor})` (rubber-band/DDA: eases up when `state.performance`>0.5, down when struggling) · `difficulty.compose(...curves)` (multiply curves, e.g. a rising wave floor plus rubber-banding). Deterministic and node-safe; per-mul weights make HP/damage climb harder than speed/rate, all clamped to `cap` (default 3).

## Steering (reusable component — engine/control/steering.js)
Low-level STEERING primitives that a behavior composes (behaviors.js is the high-level brain; these are its atoms). Each returns a desired velocity vector `[x,0,z]` from pure vector math — no side effects (except `wander`, which advances a heading on the state you pass). Blend several with `combine`, write the result to `e.vel`, and let movement.js integrate it. `steering.seek(fromPos,toPos,speed)` (go there) · `steering.flee(fromPos,fromTarget,speed)` (mirror of seek) · `steering.arrive(fromPos,toPos,speed,slowRadius=3)` (ease to a stop on the goal) · `steering.pursue(self{pos,vel},target{pos,vel},speed)` (lead a moving target) · `steering.evade(self,target,speed)` (dodge the interception) · `steering.separation(selfPos,neighborsPosArray,radius,speed)` (anti-crowding push) · `steering.wander(state,{speed,jitter,turn})` (smooth roam; pass `state.rng` for determinism) · `steering.combine([{v,weight}...],maxSpeed)` (weighted blend, length-clamped). Typical use inside a behavior: `const v = steering.combine([{v:steering.seek(e.pos,t.pos,4),weight:1},{v:steering.separation(e.pos,near,2,4),weight:1.5}], 4); e.vel[0]=v[0]; e.vel[2]=v[2];`.

## Projectile motion (reusable component — engine/control/projectiles.js)
The flight path of a bullet AFTER it is fired — the complement to weapons.js (which owns when/pattern). A game runs a motion stepper over the `bullets` registry each update (a game-wide `ctx.projectile`, or `b.motion` per bullet for mixed ammo). Each factory returns `step(bullet, dt, ctx)` that mutates `bullet.vel` and/or `bullet.pos`; per-bullet state lives on `b._m`, so one instance drives the whole registry. The movement system already integrates vel→pos, so these STEER velocity and let it fly. `ctx.target(bullet) -> nearest enemy {pos}` is optional guidance (always null-guarded); `ctx.owner`/`b.owner` gives boomerang a live homing point. Variants: `straight()` (no-op, constant velocity — the default) · `homing({turn=4,speed})` (rotate toward the nearest enemy by a max turn rate, holding speed; flies straight with no target) · `boomerang({range=8,speed})` (fly out to `range` from the launch point, then curve back to the owner and self-destruct on arrival) · `wave({amp=3,freq=6})` (sine-weave perpendicular to the launch heading — snaking, hard to line up) · `arc({gravity=30})` (lob: pile downward accel onto vel[1]; launch with vel[1]>0 for grenades/mortars/arrows) · `accelerate({accel=40,max})` (ramp ground-plane speed along the heading; negative `accel` decays to a stop without reversing, `max` caps it). Pure and sim-testable with a fake bullet.

## Emitter patterns (reusable component — engine/control/patterns.js)
The TIME-driven bullet-hell brain for a hostile emitter (enemy/boss turret) — the danmaku counterpart to weapons.js (which is the player's INPUT-driven gun). A game gives an emitter its own instance and steps it each frame, spawning whatever it returns: `e.pattern = patterns.spiral({count:4}); for (const s of e.pattern(dt, {aimDir})) spawnInto(bullets, ...)`. Each factory returns a stateful `step(dt, ctx) -> shots[]` where a shot is `{dir:[x,z], speed?}` (unit XZ vector) and `ctx.aimDir` (unit vector toward the player) steers the tracking variant. Variants: `ring({count,cooldown,offset})` (evenly-spaced full-circle burst on a cooldown) · `spiral({count,turn,rate})` (rotating spokes; base angle steps `turn` radians per fire — the classic boss curl, negative `turn` reverses) · `aimedSpread({count,arcDeg,cooldown})` (a fan centered on `ctx.aimDir`, tracks the player; straight +Z with no aim) · `spinner({arms,rate,spin})` (continuous pinwheel sweeping `spin` rad/s in real time, decoupled from fire cadence) · `pulse({count,every,pulses,gap,twist})` (a heartbeat salvo of `pulses` quick twisting rings then a long `every`-second lull). All per-instance state (angles/timers) lives in the closure so each emitter needs its own instance; pure, deterministic, node-safe. Dir convention `[sin a, cos a]` matches weapons.radial and behaviors' facing math, so shots drop straight into the same fire/spawn glue.

## Targeting (reusable component — engine/control/targeting.js)
Target SELECTION strategies — the "which one do I pick?" brain shared by aim assist, homing bullets, and enemy fire. Each factory returns a selector `select(fromPos, candidates, ctx) -> entity | null`, where `fromPos` is the [x,y,z] doing the choosing, `candidates` is EITHER a plain array of entities OR a registry-like with `.each` (e.g. `ctx.registries.enemies`), and each entity has `.pos` (+ optional `.hp`). All variants skip `.dead` entities and return null on empty/null input. Pure and deterministic given `ctx.rng` (no world/DOM/timers/Math.random), so each sim-tests against a fake array of `{pos,hp}`. Variants: `targeting.nearest()` (closest — default for aim/homing) · `farthest()` (snipers/artillery) · `lowestHp()` (finish the weak — focus fire; missing hp = Infinity) · `highestHp()` (chip the tank/boss) · `random({rng})` (uniform reservoir pick via `rng` arg or `ctx.rng`; first candidate when no rng) · `inCone({dir,arcDeg,range})` (nearest inside an aim cone; `dir` XZ vector overridable per-call via `ctx.aimDir`) · `mostClustered({radius})` (target where the swarm is densest — AoE/rockets) · `leastRecent()` (round-robin across the swarm via a per-selector WeakSet that clears after a full round). Reuse ONE selector instance to preserve round-robin/rng state; selectors never mutate the candidates.

## Status effects (reusable component — engine/control/statuses.js)
Timed buffs/debuffs that ride ON an entity (in `e._status`), swappable like hazards/pickups and decoupled from their cause — a weapon, hazard zone, pickup, or behavior can all `attach` the same status. Each factory returns a per-instance status `{ kind, apply(e), tick(e,dt) -> { damage?, speedMul?, stunned?, dmgMul?, expired } }`; the verdict is PURE DATA (the status never mutates the entity — the caller subtracts `damage`, scales speed by `speedMul`, skips input while `stunned`, scales outgoing damage by `dmgMul`). Helper `attach(e, status)` runs the one-time `apply` and pushes onto `e._status`; `step(e, dt)` ticks ALL of an entity's statuses, drops the expired in place, and folds survivors into one aggregate verdict (`damage` summed · `speedMul`/`dmgMul` multiplied · `stunned` OR'd). Attach a FRESH status per victim (state lives in its closure). Variants: `burn({dps=8,duration=3})` (fire DoT, dt-scaled) · `slow({mult=0.5,duration=2})` (speed debuff, stacks multiplicatively) · `freeze({duration=1.5})` (speedMul 0 + stunned — hard CC) · `poison({dps=4,duration=5,stacks})` (stacking DoT; re-apply adds a stack + retimes) · `stun({duration=1})` (lockout only, momentum still slides) · `haste({mult=1.5,duration=4})` (speed buff, cancels against slow) · `weaken({mult=0.5,duration=3})` (softens the entity's OWN outgoing damage via dmgMul). Node-safe and sim-testable with a fake entity and dt stepping.

## Damage resolution (reusable component — engine/control/damage.js)
The layer between "a hit landed" and the raw hp subtraction the health system does. A game composes MODIFIERS into one policy and asks what a hit is worth: `resolve(base, cfg, ctx) -> { amount, crit, knockback:[x,z]|null, tags, heal, pierce }`. `base` is the nominal damage (weapon/enemy stat), `cfg` is a composed policy (or an array / single modifier / null), `ctx` carries the encounter `{ from|attacker, to|target, rng }` (positions as entities or [x,y,z]/[x,z]). The verdict is PURE DATA — resolve never mutates entities; the caller subtracts `amount`, adds `knockback` to the target's vel, heals the attacker by `heal`, and keeps the projectile alive for `pierce-1` more hits. Modifiers: `crit({chance=0.15,mult=2})` (rng-gated multiplier — draws ctx.rng, no-op without it) · `armor({flat=0,resist=0})` (subtract flat then scale by 1-resist, floored at 0) · `knockback({force=6})` (unit push attacker->target * force) · `falloff({near=0,far=20,min=0})` (full within near, fading to min fraction at far) · `lifesteal({frac=0.5})` (report heal = frac * dealt; place LAST so it sees armor) · `pierce({count=1})` (targets one hit passes through). `compose(...mods)` folds them left-to-right (flattens arrays, skips falsy so `compose(a, cond && b)` is safe) into one policy: `ctx.damage = damage.compose(damage.crit(), damage.armor({resist:0.2}))`. Put multipliers (crit/falloff) before reducers (armor). Deterministic given ctx.rng; node-safe (no globals/timers/Math.random), so a sim seeds the rng and asserts exact numbers.

## Abilities (reusable component — engine/control/abilities.js)
Activatable player POWERS with cooldowns + durations — the discrete, button-driven counterpart to movements.js (continuous stick locomotion). Give the player ONE instance per ability (state is per-instance in the closure) and wire it to a button. Each factory returns `{ ready(), trigger(entity, dir, ctx) -> bool, tick(entity, dt) -> state }`: `ready()` asks if the cooldown is spent; `trigger(e, dir, ctx)` fires the effect only when ready (applies it, starts the cooldown, returns true; a no-op false while cooling); `tick(e, dt)` advances the cooldown + active timers each frame, runs per-frame effects, and returns `{ ready, cooldown, active, ... }` for the HUD/loop. `dir` is an XZ aim vector `[x,z]` (falls back to entity facing `rot`). Variants: `dash({distance=6,cd=1,dashTime=0.15})` (velocity burst along dir; drives e.vel while active so it overrides movement) · `dodgeRoll({distance=5,iframes=0.4,cd=0.8})` (dash that also opens an `e.invuln` i-frame window the health system checks) · `blink({range=8,cd=3})` (instant `e.pos` teleport by dir*range) · `shieldAbility({duration=3,cd=6})` (sets `e.shield` seconds-remaining timer) · `groundPound({cd=2,radius,force,damage})` (marks a `{pos,radius,force,damage}` shockwave request the game reads ONCE off `tick().shockwave`) · `timeSlow({factor=0.4,duration=2,cd=8})` (reports `state.timescale` 0..1 the loop multiplies into sim dt; tick with REAL dt). Pure, deterministic, node-safe. INTEGRATION: tick abilities AFTER movement.js so a dash's velocity wins for its window; effects surface as plain entity fields (`e.invuln`, `e.shield`) or tick state (`shockwave`, `timescale`) that a tiny game system / the loop applies.

## Vehicles (reusable component — engine/control/vehicles.js)
Set `ctx.movement` in bootstrap to drive a VEHICLE instead of on-foot locomotion — same `update(entity, input, dt)` shape as movements.js (`input = {move:[x,z], aim:[x,z], jump}`) but with car/boat/plane physics feel. Heading (`e.rot`) and velocity (`e.vel`) persist on the entity across frames (convention: forward = `[sin(rot),cos(rot)]`, throttle = `-move[1]`, matching movements.tank); the movement system integrates vel→pos as usual. Variants: `vehicles.car({accel=20,topSpeed=16,grip=6,turn=2.6})` (throttle on move[1], steering scales with speed so it can't pivot in place, lateral grip bleeds off skid so hard turns drift; low grip = slidey) · `vehicles.boat({accel=8,drag=1.2,turn=1.4})` (heavy momentum, omni water drag glides it to a stop, rudder bites only underway) · `vehicles.hover({accel=18,drag=3})` (omni-directional thrust with glide inertia — no fixed nose, faces its travel) · `vehicles.flyer({thrust=14,lift=10,turn=2})` (arcade flight: move[1] thrust along the nose, move[0] banks the heading, aim[1] climbs/dives via vertical velocity, light air drag, roll hint on `e._veh.bank` for a renderer to tilt the mesh) · `vehicles.heavyTank({accel=10,turn=1.2})` (pivots in place, slow inertial accel, heavy tread friction to a halt when you let off the gas). Pure and node-safe (no window/DOM/timers/Math.random); per-vehicle bank scratch lives on `e._veh`, so it sim-tests with a fake entity and dt stepping.

## Formations (reusable component — engine/control/formations.js)
Squad/escort/wave POSITIONING — the "where does each follower stand relative to the leader?" brain, swappable like behaviors/steering. Pick one per squad and ask it for a slot each frame: `const fmt = formations.wedge({ gap: 2 }); const target = fmt(i, squad.length, leader.pos, leader.rot); const v = steering.arrive(e.pos, target, e.speed); e.vel[0]=v[0]; e.vel[2]=v[2];`. Each factory returns a PURE `slot(index, count, leaderPos:[x,y,z], leaderRot) -> [x,0,z]` world target; slots are computed in the leader's local frame (local +z = forward, +x = right) then rotated by `leaderRot`, so the whole shape turns to face where the leader faces. No state, no globals — one instance drives the whole squad, and followers just `steering.arrive()` to their slot. Variants: `line({gap=2})` (side-by-side rank centered on the leader — a shield wall/firing line) · `column({gap=2})` (single file directly behind — a convoy/snake) · `wedge({gap=2,angleDeg=45})` (V/arrowhead with the leader at the tip, slots alternating to left/right flanks one rank deeper per pair — a spearhead) · `circle({radius=4})` (even ring around the leader, slot 0 at the front — a bodyguard cordon) · `grid({cols=3,gap=2})` (rows×columns block trailing back, columns centered — a phalanx) · `echelon({gap=2})` (diagonal stagger, each unit one step back AND to the flank — a slanted strafing line). Deterministic and node-safe; sim-tests by asserting slot geometry (centering, symmetry, ring radius, rotation of the whole formation).

## Pathing (reusable component — engine/control/pathing.js)
Route FOLLOWING and simple grid navigation — one rung ABOVE the steering primitives (steering.js) and a sibling of behaviors.js. Route followers return a stepper `step(entity, dt) -> [x,0,z]` (the desired velocity, also written to `e.vel` + facing `e.rot`), with per-entity progress kept on `e._path`, so ONE instance drives a whole registry. Variants: `pathing.followPath({points:[[x,z],...], loop=true, arrive=0.4, speed})` (walk a fixed route waypoint by waypoint; advance within `arrive`, wrap when `loop` else park on the last point) · `pingPong({points, arrive, speed})` (bounce back and forth end-to-end forever; with 2 points a simple shuttle) · `patrolPath({points, wait=0, loop=true, arrive, speed})` (followPath that PAUSES `wait` seconds at each stop — sentries; `wait` is a number or a per-stop array) · `lerpTo({speed=6, arrive})` (ease toward a MOVING goal exponentially and frame-rate-independently — soft-follow companion / camera dolly; call as `step(e, goalPos, dt)`, accepts `[x,y,z]` or `[x,z]`). `gridNav({cell=1, passable(cx,cz)->bool})` is the odd one out — a PURE greedy 8-direction planner with NO world access: you hand it a `passable` predicate over integer cell coords and call `nav.stepToward(fromPos, goalPos) -> next [x,z]` (the center of the best open neighbour toward the goal, skipping blocked cells and corner-cutting diagonals, holding position when boxed in). It also exposes `toCell`/`toWorld` for aligning entities to the grid. Chain `stepToward` to crawl a route one cell at a time — cheap for swarms on mobile, no full A* alloc. All node-safe (no window/DOM/timers/Math.random) and sim-testable with a fake entity.

## Camera effects (reusable component — engine/control/camerafx.js)
Wrappers that DECORATE a base camera from cameras.js. Each is `fx(base, cfg) -> (focus, entity, dt) -> {eye,target,...}` — same shape as a raw camera, so they compose and pass `up`/`projection`/`orthoSize` straight through: `ctx.camera = cameraFx.smooth(cameraFx.shake(cameras.topDown()))`. Variants: `shake(base,{mag=0.5,decay=5,freq})` (decaying trauma jitter on the eye; fire it with the returned `.trigger(mag)` on a hit/explosion, or set `entity.shake`; seeded counter = deterministic replays) · `smooth(base,{stiffness=8})` (frame-rate-independent exponential ease of eye+target toward the base — kills snap/jitter; first frame latches) · `lookAhead(base,{lead=1.2,includeEye})` (offset target/eye by `entity.vel*lead` so the camera leads a moving subject) · `zoom(base,{min=1,max=1.6,on})` (dolly eye distance from target by a factor blended `min..max` via `on(entity)->t`, e.g. speed/health) · `deadzone(base,{radius=2,radiusZ})` (hold a lagged anchor until the focus leaves a slack box, then snap to the edge — no wobble from tiny moves). Pure, node-safe, per-instance state in closures.

## Tweening + interpolation (reusable component — engine/control/tweens.js)
The pure math of "animate a value from A to B over `duration`, shaped by an easing curve" — a value PRODUCER a game reads each frame (popup scale, fade alpha, color blend, camera dolly); it never draws or touches the DOM. `tweens.tween(from, to, duration=1, ease='linear')` returns a stepper `step(dt) -> { value, done, t }` with all progress captured in the closure (reuse ONE per animation): `value` is the eased interpolation, `t` is RAW linear progress 0..1 (assert monotonic/endpoints), `done` latches at t=1; `duration<=0` snaps instantly; `step.reset()` rewinds and `step.duration` exposes the length. A tweened value is a NUMBER or an ARRAY (lerped componentwise — positions/colors/offsets). Easing curves live under `tweens.ease`: `linear` · `easeIn` (quad accel) · `easeOut` (quad decel) · `easeInOut` · `easeOutBack` (overshoot then settle) · `easeOutElastic` (springy) · `easeOutBounce` · `smoothstep` (Hermite S-curve) — pass a curve name or your own `t->t` fn; all land exactly on 0 and 1 at the ends even when they leave the unit range mid-flight. Helpers: `tweens.lerp(a,b,t)` (number or array, extrapolates past 0..1) · `tweens.damp(cur,target,lambda=8,dt)` (frame-rate-INDEPENDENT exponential smoothing — the correct replacement for `cur+=(target-cur)*k`; number or array) · `tweens.pingPongT(t)` (fold a rising clock onto a 0->1->0 triangle wave of period 2) · `tweens.sequence([tweens])` (play tweens back to back as one animation, same `step(dt)->{value,done,t}` shape with overall `t`; time is accounted globally so long frames spill accurately into the next segment, and sequences NEST). Pure, deterministic, node-safe (no Math.random/timers/DOM/globals), sim-tested by stepping and asserting curve shape (endpoints, monotonicity, overshoot, spillover).

## Screen/game feel effects (reusable component — engine/control/effects.js)
Screen/game FEEL signals — the "juice" a renderer or the loop reads each frame. These are pure STATE MACHINES that PRODUCE values (a shake offset, a timescale, an overlay alpha, a zoom factor); they never draw or touch WebGL/DOM, and use a seeded counter instead of Math.random so replays are deterministic. Keep one instance per effect, fire it on an event, and sample it per frame. Every factory returns the same tiny shape: `{ trigger(cfg?), step(dt) -> value, active }` — `trigger` starts/restarts (optional per-fire cfg overrides), `step(dt)` advances and returns this frame's value, `active` reports whether it's still running. Variants: `screenShake({mag=0.6,duration=0.4,decay=6})` (`step -> [x,y]` decaying wobble on a seeded counter to add to the camera eye; linear timer × exponential settle; stacking takes the stronger shake) · `hitStop({duration=0.06})` (`step -> timescale` 0 while frozen else 1 — multiply into sim dt for a crunchy impact pause; tick with REAL unscaled dt; re-trigger extends) · `flash({color=[1,1,1],duration=0.2,peak=1})` (`step -> alpha 0..1` for a full-screen color overlay; carries `.color` for the renderer to tint, recolorable per trigger) · `zoomPunch({amount=0.15,duration=0.25,attack=0.25})` (`step -> zoom multiplier` around 1 that spikes then smoothstep-eases back; >1 punches in, negative `amount` punches out) · `vignettePulse({duration=0.6,peak=1,hold=0})` (`step -> alpha 0..1` edge-darkening throb that rises fast / ebbs slow, with optional `hold` seconds pinned at peak for stateful glows). Per-instance closure state, node-safe, sim-testable by triggering then stepping dt. Typical wiring: `const shake = effects.screenShake(); on('enemy-died', () => shake.trigger()); // render: const [ox,oy]=shake.step(dt); eye[0]+=ox; eye[2]+=oy;`

## Particles (reusable component — engine/control/particles.js)
CPU particle EMITTER logic — a pure DATA sim the renderer later draws (no WebGL, no DOM here; your component only PRODUCES the particle array a game hands to its point/quad drawer). A particle is `{ pos:[x,y,z], vel:[x,y,z], life, age, size, color:[r,g,b] }`. Make ONE system per effect budget: `const fx = particles.makeSystem({ gravity=-9.8, drag=0, max=2000, rng })` → `{ particles, count, emit(spec,ctx), step(dt), clear() }`. `step(dt)` ages every particle (`age+=dt`), adds gravity to `vel[1]`, applies per-second linear `drag`, integrates `pos+=vel*dt`, and drops those with `age>=life` (compacting `particles` in place — the array ref is stable). A SPEC is a stateful fn `(add, ctx) => void` that calls `add({pos?,vel?,life?,size?,color?,g?,drag?})` once per new particle (`g` overrides system gravity, e.g. sparks); `ctx = { pos, dir, dt, rand, rng }` (origin defaults to `[0,0,0]`). Preset emit-spec factories: `burst({count=16,speed=6,life=0.6,spread=Math.PI,drag=1.5})` (one-shot omni explosion pop, brakes to a hang) · `sparks({count=12,speed=7,gravity=22})` (one-shot upward-biased shards that rain back down — impacts/muzzle grit) · `fountain({rate=30,up=8})` (continuous upward jet, `rate` particles/sec, arcs down) · `stream({rate=40,dir=[0,0,1],speed=10})` (continuous directed jet; pass `ctx.dir` to swivel — thruster/flamethrower) · `trail({life=0.4,every=0})` (one near-still fading breadcrumb per call AT `ctx.pos`; `every` seconds throttles). Rate-based presets keep a fractional carry in their closure, so give each emitter its OWN instance and call it every frame with `ctx.dt`. Deterministic given `ctx.rng` (or an internal phase fallback) — no Math.random, no globals, node-safe and sim-testable (emit N, step, assert aging/expiry/counts). `particles` bundle = `{ makeSystem, burst, fountain, stream, trail, sparks }`.

## Transitions (reusable component — engine/control/transitions.js)
Screen/scene TRANSITIONS as time-based PROGRESS producers a renderer or DOM overlay reads — nothing here draws (no WebGL/DOM/timers/Math.random). Each factory returns a stateful stepper `step(dt) -> { p (0..1 eased progress), done, ...visual }`; the extra fields are exactly what the overlay paints that frame. Every stepper also carries `step.replay()` (restart), `step.reverse()` (flip direction — out<->in — and keep going), `step.reset()` (alias), and mirrors the latest `{ done, p }` onto the function itself. Variants: `fade({duration=0.5,dir="out"}) -> {alpha}` (fade TO black when dir="out", FROM black when "in") · `wipe({duration,axis="x"}) -> {p,axis}` (a hard bar sweeping across) · `slide({duration,from="left"}) -> {offset:[x,y]}` (normalized -1..1 screen displacement of a pushed panel; left/right/up/down) · `iris({duration,dir="open",center}) -> {radius 0..1}` (circular mask; "open" reveals 0->1, "close" is the cartoon iris-out 1->0) · `crossfade({duration}) -> {a,b}` (outgoing/incoming layer weights, a+b===1) · `pixelate({duration,dir="out",max=64,min=4}) -> {amount,blocks}` (chunk the frame down) · `none()` (instant cut placeholder). All accept `delay` and `easing` (from the exported `ease` set). Compose a full scene change with `pair(outT, inT, { onSwap }) -> step(dt) -> { p, phase:'out'|'in'|'done', done, out, in }`: it runs `outT` to completion, fires `onSwap` ONCE (swap scenes / tear down the old level here), then runs `inT`, reporting one blended progress (out half 0..0.5, in half 0.5..1); `reverse()` runs the whole thing backwards. Also exports `timeline` (the shared play head), `ease` (linear/quad/cubic/smoothstep/…), `lerp`, `clamp01`. Pure, deterministic, node-safe — progress is driven only by the fed `dt`, so a sim steps to completion and asserts endpoints + monotonic progress; per-instance state (elapsed, dir) lives in closures.

## Palettes (reusable component — engine/control/palettes.js)
COLOR + PALETTE helpers for the low-poly look (engine colors are `[r,g,b]`, each channel 0..1 — never 0..255). Two halves: NAMED RAMPS and HELPERS, all pure/deterministic (no Math.random/DOM/timers; variation comes from a passed rng or an index) and non-mutating (every helper returns a fresh array, so swatches pulled from a shared palette are safe to reuse). `palettes.PALETTES` is a table of 8 mood ramps (5 swatches each, ordered darkest→brightest): `sunset · neon · forest · mono · pastel · retro · ice · lava`; `palettes.palette(name)` returns a defensive COPY (falls back to mono on an unknown name). Channel math: `lerpColor(a,b,t)` (clamped per-channel blend) · `lighten(c,amt)`/`darken(c,amt)` (toward white/black) · `saturate(c,amt)` (amt -1..1, holds luma; -1 = full gray) · `grayscale(c)` (Rec.601 luma). Hue: `hueShift(c,deg)` (rotate hue via HSL, wraps at 360 any sign, holds S+L). Feedback + procedural: `damageFlash(base,t,{peak})` (hit tint — white-blown at t=0, quadratic-eases back to base by t=1, auto-clamped) · `teamColor(index,{sat,light,count})` (distinct hue per player/faction — hand-picked 8-hue wheel for small rosters, or evenly-spaced `count` hues for big lobbies; index wraps). Ramp sampling: `gradient(pal,t)` (sample a name-or-array palette as a continuous 0..1 ramp, lerp between neighbours) · `pick(pal,sel)` (one swatch; `sel` is an rng `{int|next}` for a uniform pick or a wrapping numeric index, -1 = last). Hex I/O: `toHex(c)` → `'#rrggbb'` (clamps+rounds) · `fromHex(str)` accepts `#rgb`/`#rrggbb` with or without `#` (bad input → black). A game reads results straight into `e.color` / mesh tints — e.g. `enemy.color = palettes.pick('lava', rng)`, level fog `palettes.gradient('ice', depth)`, hit flash `e.color = palettes.damageFlash(e.baseColor, e.flashT)`, player tint `palettes.teamColor(playerIndex)`. Node-safe and sim-testable (assert channel math, palette shape, hue wrap).

## Building a game from a description (the app's core job)
The app's purpose is making games. Given a description, DON'T write an engine — pick components and glue:
1. **View + movement** from the description: top-down shooter → `cameras.topDown` + `movements.twinStick`; platformer → `cameras.sideScroller` + `movements.platformer`; runner → `cameras.sideScroller` + `movements.autoRun`; true 2D → `cameras.flat2D`.
2. **Systems** in `game/bootstrap.js`: keep the ones the game needs (movement, ai, collision, health, score, hud, audio, render, gamestate are almost always in; add `fire`/`spawn` for shooters, drop them for a pure platformer). For a shooter also pick `ctx.weapon` (single/rapid/shotgun/burst/radial) and `ctx.aim` (stick/facing/manual/autoAim) — e.g. a one-thumb mobile shooter is `aim.autoAim` + `weapons.rapid`. Pick enemy/NPC brains with `ctx.behavior` (chase/orbit/zigzag/charger for foes; wander/patrol/follow/guard for NPCs), or `behaviors.byKind({...})` to mix.
3. **Entities** (`game/entities.js`): mesh (box/sphere/cylinder/plane), color, size, stats — pure data. **Config** (`game/config.js`): the numbers.
4. **Signals/registries** (`game/contracts.js`): add any the game needs.
5. Ask the user AT MOST 1–2 questions only for genuinely ambiguous core choices (e.g. "waves or endless? health or one-hit?"). Otherwise pick sensible defaults and build.
6. Only write a NEW system for behavior no component provides — one small file, imports only core, reacts to events/signals.

You should not need to open `engine/core`, `engine/render`, `engine/control`, or the existing `engine/systems`.
