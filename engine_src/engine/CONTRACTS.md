# Engine contracts â€” the whole interface

This is the only engine file you normally need to read. The engine is a black
box; you build a game by writing `game/` (glue), not by editing `engine/`.

## The verbs (import from engine/core)
- `signal(v)` â†’ `{ get(), set(v), update(fn), subscribe(fn) }` â€” one reactive value.
- `makeRegistry()` â†’ `{ add(e), remove(e), each(fn), query(pred), has(e), size, flush(), clear() }` â€” a set of entities; add/remove are deferred (safe to mutate while iterating).
- `on(name, fn)` / `off(name, fn)` / `emit(name, data)` â€” the event bus. `on` returns an unsubscribe fn.
- `spawn(kind, opts)` / `spawnInto(reg, kind, opts)` / `despawn(e)` â€” entities. `despawn` removes from the scene AND the entity's registry.
- `makeRng(seed)` â†’ `{ next(), range(a,b), int(a,b) }` â€” deterministic RNG.
- `makeLoop({ paused, registries })` â†’ `{ start(), stop(), step(dt) }` â€” the frame driver.

## Entity fields
`{ kind, pos:[x,y,z], rot, scale (number or [x,y,z]), vel:[x,y,z], color:[r,g,b] (0..1), mesh, radius, dead, ...your fields }`.
`mesh` is one of: `box | sphere | cylinder | plane`. `spawn(opts.at)` sets pos.

## Phases (emitted by the loop each frame, in order)
`input â†’ update(dt) â†’ physics â†’ late` (the sim; skipped when paused) then `render(dt)` (always).
Continuous state â†’ **signals** (input sticks). Discrete happenings â†’ **events**.

## Signals the systems expect (define these in game/contracts.js)
`gameState` (menu|playing|paused|over) Â· `score` Â· `wave` Â· `playerHealth` Â· `moveInput` [x,z] Â· `aimInput` [x,z] Â· `firing` Â· `player` (the entity).

## Registries the systems expect
`enemies` Â· `bullets`.

## Events (the vocabulary)
Raised by systems: `game-started` Â· `game-over` Â· `player-fired` Â· `enemy-hit {enemy,bullet}` Â· `enemy-died {enemy}` Â· `player-hit {enemy}` Â· `player-damaged {hp}` Â· `player-died` Â· `wave-started {n}` Â· `wave-cleared {n}`.

## Systems (engine/systems/*, each imports only core)
`input` (touchâ†’sticks) Â· `movement` (input+velocityâ†’position) Â· `ai` (runs each enemy/NPC's behavior) Â· `fire` (spawn bullets) Â· `spawn` (wave director) Â· `collision` (overlapâ†’events) Â· `health` (hitsâ†’damage/death) Â· `score` Â· `cleanup` (life/bounds despawn) Â· `gamestate` (lifecycle+pause) Â· `hud` (DOM overlay) Â· `audio` (WebAudio SFX) Â· `render` (WebGL follow-cam).

## Cameras (reusable component â€” engine/render/cameras.js)
Set `ctx.camera` in bootstrap to pick a view; all are swappable without touching game logic:
`cameras.topDown({height,back})` Â· `cameras.sideScroller({distance,height,ahead})` Â· `cameras.thirdPerson({distance,height})` Â· `cameras.orbit({radius,height,speed})` Â· `cameras.fixed({eye,target})` Â· `cameras.flat2D({height,size})` (true 2D, orthographic top-down) Â· `cameras.sideScroller2D({distance,size})` (2D platformer). Each is a pure `(focus, entity, dt) â†’ {eye,target,up?,projection?,orthoSize?}`; add your own the same way.

## Movement (reusable component â€” engine/control/movements.js)
Set `ctx.movement` in bootstrap: `movements.twinStick({speed})` (top-down) Â· `movements.eightWay({speed})` Â· `movements.tank({speed,turn})` Â· `movements.platformer({speed,jump,gravity})` (side view, gravity+jump) Â· `movements.autoRun({speed,jump})` (endless runner). Each is `update(entity, input, dt)` where `input = {move:[x,z], aim:[x,z], jump}`. Default is twinStick.

## Weapons (reusable component â€” engine/control/weapons.js)
Set `ctx.weapon` in bootstrap to pick the gun (rhythm + pattern): `weapons.single({cooldown})` (default) Â· `weapons.rapid({cooldown})` (machine gun) Â· `weapons.shotgun({pellets,spreadDeg,cooldown})` (fan) Â· `weapons.burst({count,gap,cooldown})` (tap-fire) Â· `weapons.radial({ways,cooldown})` (nova/bomb). Each is a stateful `update(dt, firing, dir) â†’ shots[]`; a shot is `{dir:[x,z], speed?, life?, damage?, scale?}`. The fire system spawns whatever it returns, so this is the one place that decides "what a shot is." Per-shot `damage` overrides `config.bulletDamage`.

## Aim (reusable component â€” engine/control/aim.js)
Set `ctx.aim` in bootstrap to pick how shots are pointed: `aim.stick()` (twin-stick, default) Â· `aim.facing()` (shoot where you move â€” one-stick/auto-fire) Â· `aim.manual()` (sticky: flick to set, holds after release) Â· `aim.autoAim({range})` (lock the nearest enemy â€” mobile aim assist). Each is `resolve(player, raw, ctx) â†’ [x,z]` (a unit direction). The fire system asks the aimer each shot.

## Behaviors (reusable component â€” engine/control/behaviors.js)
The brain for each enemy/NPC (in the `enemies` registry). Set a game-wide default `ctx.behavior`, or give one entity its own `e.behavior`, or mix types with `behaviors.byKind({kind: ..., default: ...})`. Enemy: `chase()` (default) Â· `flee()` Â· `orbit({radius,dir})` (circle-strafe) Â· `keepDistance({min,max})` (ranged) Â· `zigzag({amp,freq})` (weaves) Â· `charger({range,windup,dashSpeed})` (telegraphâ†’dash). NPC: `wander({area})` (roam near spawn) Â· `patrol({points})` (waypoint route) Â· `follow({distance})` (companion) Â· `guard({radius})` (hold home, chase intruders). Each is `step(entity, target, dt, ctx)` and sets velocity + facing; per-entity state lives on `e._ai`, so one instance drives a whole registry. `target` is the player, or null when it's gone (NPC behaviors keep going; chasers idle).

## Spawn directors (reusable component â€” engine/control/spawners.js)
The "when and what to spawn" brain, swappable like cameras/movements/behaviors. Set `ctx.director` in bootstrap; the spawn system calls it each update phase as `director(dt, api)`. A director is a PURE stepper â€” it never imports world/rng, it only drives the injected `api = { count() (live enemy count), spawn(opts), setWave(n), rng, onWave(cb) }`, so it unit-tests with a recording fake. Variants: `waves({base,growth,delay})` (escalating arena rounds; advances to the next, bigger wave once `count()===0` and a `delay` breather passes â€” batch n = base+growth*(n-1)) Â· `endless({rate,max})` (steady time-based trickle, catches up long frames, `max` caps live count) Â· `burst({size,every,max})` (periodic clumps with lulls) Â· `boss({atWave,make,escort})` (watches the wave via `onWave`, spawns one boss + escorts once, then steps aside) Â· `timed({schedule:[{at,count,make}],loop})` (scripted absolute-time cues for intros/boss-rush). Compose with `chain(...directors)` â€” it forwards any wave announced via `setWave` into every child's `onWave`, so `chain(waves(), boss({atWave:5, make}))` lets a boss watch a wave loop. `spawn(opts)` shape is the game's own contract (opts carries { wave } | { pack } | { boss, make, escort } | { make, at }).

## Pickups (reusable component â€” engine/control/pickups.js)
What an item does when the player collects it â€” a swappable effect component. Set it per-drop on an item entity (`item.pickup = pickups.health({ amount: 50 })`), not game-wide. Each factory returns `{ kind, apply(ctx, player) }`; `apply` mutates the game's signals and/or stashes a timed buff on the player, then returns a short tag string ('heal'|'shield'|'score'|'speed'|'weapon'|'magnet'|'life') for a label/sound. Variants: `health({amount,cap})` (raise playerHealth, capped) Â· `shield({duration})` (player.shield invuln timer) Â· `scoreBonus({points})` Â· `speedBoost({mult,duration})` (player.speedMult+speedTimer for movement) Â· `weaponSwap({weapon})` (set ctx.weapon) Â· `magnet({radius,duration})` (player.magnetRadius+magnetTimer) Â· `extraLife({refill,cap})` (bump lives signal/player.lives, optional heal). Helper `makePickup(kind, apply)`. Timed buffs are seconds-remaining fields a tiny game system counts down and clears.

## Hazards (reusable component â€” engine/control/hazards.js)
Environmental effect zones â€” lava, spikes, wind, pits â€” swappable like behaviors. A game places a zone in the world as either a circle `{pos:[x,y,z], radius}` or an axis-aligned box `{min:[x,z], max:[x,z]}`, then picks a hazard per zone: `const lake = hazards.lava()`. Each hazard exposes `inside(entity, zone) -> bool` (radius-aware membership, works for both shapes) and `affect(entity, dt, zone) -> {damage?, push?[x,z], kill?, slow?} | null`. The verdict is PURE DATA â€” the hazard never mutates the entity; a system reads the fields and applies them (subtract `damage`, add `push` to vel, despawn on `kill`, scale speed by `slow` 0..1). Returns null when the entity is outside. Variants: `damageZone({dps=10})` (dt-scaled DoT) Â· `spikes({damage=20})` (one-shot flat hit on entry, re-arms after you leave â€” per-entity via WeakSet) Â· `lava({dps=30,slow=0.5})` (DoT + drag) Â· `slowField({mult=0.5})` (pure speed multiplier â€” mud/web) Â· `pit({})` (instant kill inside) Â· `windZone({force:[x,z]})` (constant dt-scaled velocity push â€” conveyor/fan/current; also exposes `outward(entity,zone)` for radial repulsors). Pure and sim-testable; a game applies verdicts wherever it steps hazards (e.g. a small system in `update`).

## Difficulty (reusable component â€” engine/control/difficulty.js)
Set `ctx.difficulty` in bootstrap to pick how the game ramps up. Each is a pure `scale(state) -> {speedMul, hpMul, rateMul, damageMul}` where `state = {time, score, wave, performance}`; the spawn director multiplies enemy stats by the muls when building a wave. `difficulty.flat({speed,hp,rate,damage})` (fixed, a casual baseline) Â· `difficulty.linear({per,cap,by,unit})` (smooth continuous ramp vs time/wave/score, default) Â· `difficulty.stepped({every,step})` (discrete plateaus that jump) Â· `difficulty.waveBased({perWave})` (keyed to the wave counter; wave 1 = base) Â· `difficulty.adaptive({up,down,floor})` (rubber-band/DDA: eases up when `state.performance`>0.5, down when struggling) Â· `difficulty.compose(...curves)` (multiply curves, e.g. a rising wave floor plus rubber-banding). Deterministic and node-safe; per-mul weights make HP/damage climb harder than speed/rate, all clamped to `cap` (default 3).

## Steering (reusable component â€” engine/control/steering.js)
Low-level STEERING primitives that a behavior composes (behaviors.js is the high-level brain; these are its atoms). Each returns a desired velocity vector `[x,0,z]` from pure vector math â€” no side effects (except `wander`, which advances a heading on the state you pass). Blend several with `combine`, write the result to `e.vel`, and let movement.js integrate it. `steering.seek(fromPos,toPos,speed)` (go there) Â· `steering.flee(fromPos,fromTarget,speed)` (mirror of seek) Â· `steering.arrive(fromPos,toPos,speed,slowRadius=3)` (ease to a stop on the goal) Â· `steering.pursue(self{pos,vel},target{pos,vel},speed)` (lead a moving target) Â· `steering.evade(self,target,speed)` (dodge the interception) Â· `steering.separation(selfPos,neighborsPosArray,radius,speed)` (anti-crowding push) Â· `steering.wander(state,{speed,jitter,turn})` (smooth roam; pass `state.rng` for determinism) Â· `steering.combine([{v,weight}...],maxSpeed)` (weighted blend, length-clamped). Typical use inside a behavior: `const v = steering.combine([{v:steering.seek(e.pos,t.pos,4),weight:1},{v:steering.separation(e.pos,near,2,4),weight:1.5}], 4); e.vel[0]=v[0]; e.vel[2]=v[2];`.

## Projectile motion (reusable component â€” engine/control/projectiles.js)
The flight path of a bullet AFTER it is fired â€” the complement to weapons.js (which owns when/pattern). A game runs a motion stepper over the `bullets` registry each update (a game-wide `ctx.projectile`, or `b.motion` per bullet for mixed ammo). Each factory returns `step(bullet, dt, ctx)` that mutates `bullet.vel` and/or `bullet.pos`; per-bullet state lives on `b._m`, so one instance drives the whole registry. The movement system already integrates velâ†’pos, so these STEER velocity and let it fly. `ctx.target(bullet) -> nearest enemy {pos}` is optional guidance (always null-guarded); `ctx.owner`/`b.owner` gives boomerang a live homing point. Variants: `straight()` (no-op, constant velocity â€” the default) Â· `homing({turn=4,speed})` (rotate toward the nearest enemy by a max turn rate, holding speed; flies straight with no target) Â· `boomerang({range=8,speed})` (fly out to `range` from the launch point, then curve back to the owner and self-destruct on arrival) Â· `wave({amp=3,freq=6})` (sine-weave perpendicular to the launch heading â€” snaking, hard to line up) Â· `arc({gravity=30})` (lob: pile downward accel onto vel[1]; launch with vel[1]>0 for grenades/mortars/arrows) Â· `accelerate({accel=40,max})` (ramp ground-plane speed along the heading; negative `accel` decays to a stop without reversing, `max` caps it). Pure and sim-testable with a fake bullet.

## Emitter patterns (reusable component â€” engine/control/patterns.js)
The TIME-driven bullet-hell brain for a hostile emitter (enemy/boss turret) â€” the danmaku counterpart to weapons.js (which is the player's INPUT-driven gun). A game gives an emitter its own instance and steps it each frame, spawning whatever it returns: `e.pattern = patterns.spiral({count:4}); for (const s of e.pattern(dt, {aimDir})) spawnInto(bullets, ...)`. Each factory returns a stateful `step(dt, ctx) -> shots[]` where a shot is `{dir:[x,z], speed?}` (unit XZ vector) and `ctx.aimDir` (unit vector toward the player) steers the tracking variant. Variants: `ring({count,cooldown,offset})` (evenly-spaced full-circle burst on a cooldown) Â· `spiral({count,turn,rate})` (rotating spokes; base angle steps `turn` radians per fire â€” the classic boss curl, negative `turn` reverses) Â· `aimedSpread({count,arcDeg,cooldown})` (a fan centered on `ctx.aimDir`, tracks the player; straight +Z with no aim) Â· `spinner({arms,rate,spin})` (continuous pinwheel sweeping `spin` rad/s in real time, decoupled from fire cadence) Â· `pulse({count,every,pulses,gap,twist})` (a heartbeat salvo of `pulses` quick twisting rings then a long `every`-second lull). All per-instance state (angles/timers) lives in the closure so each emitter needs its own instance; pure, deterministic, node-safe. Dir convention `[sin a, cos a]` matches weapons.radial and behaviors' facing math, so shots drop straight into the same fire/spawn glue.

## Targeting (reusable component â€” engine/control/targeting.js)
Target SELECTION strategies â€” the "which one do I pick?" brain shared by aim assist, homing bullets, and enemy fire. Each factory returns a selector `select(fromPos, candidates, ctx) -> entity | null`, where `fromPos` is the [x,y,z] doing the choosing, `candidates` is EITHER a plain array of entities OR a registry-like with `.each` (e.g. `ctx.registries.enemies`), and each entity has `.pos` (+ optional `.hp`). All variants skip `.dead` entities and return null on empty/null input. Pure and deterministic given `ctx.rng` (no world/DOM/timers/Math.random), so each sim-tests against a fake array of `{pos,hp}`. Variants: `targeting.nearest()` (closest â€” default for aim/homing) Â· `farthest()` (snipers/artillery) Â· `lowestHp()` (finish the weak â€” focus fire; missing hp = Infinity) Â· `highestHp()` (chip the tank/boss) Â· `random({rng})` (uniform reservoir pick via `rng` arg or `ctx.rng`; first candidate when no rng) Â· `inCone({dir,arcDeg,range})` (nearest inside an aim cone; `dir` XZ vector overridable per-call via `ctx.aimDir`) Â· `mostClustered({radius})` (target where the swarm is densest â€” AoE/rockets) Â· `leastRecent()` (round-robin across the swarm via a per-selector WeakSet that clears after a full round). Reuse ONE selector instance to preserve round-robin/rng state; selectors never mutate the candidates.

## Status effects (reusable component â€” engine/control/statuses.js)
Timed buffs/debuffs that ride ON an entity (in `e._status`), swappable like hazards/pickups and decoupled from their cause â€” a weapon, hazard zone, pickup, or behavior can all `attach` the same status. Each factory returns a per-instance status `{ kind, apply(e), tick(e,dt) -> { damage?, speedMul?, stunned?, dmgMul?, expired } }`; the verdict is PURE DATA (the status never mutates the entity â€” the caller subtracts `damage`, scales speed by `speedMul`, skips input while `stunned`, scales outgoing damage by `dmgMul`). Helper `attach(e, status)` runs the one-time `apply` and pushes onto `e._status`; `step(e, dt)` ticks ALL of an entity's statuses, drops the expired in place, and folds survivors into one aggregate verdict (`damage` summed Â· `speedMul`/`dmgMul` multiplied Â· `stunned` OR'd). Attach a FRESH status per victim (state lives in its closure). Variants: `burn({dps=8,duration=3})` (fire DoT, dt-scaled) Â· `slow({mult=0.5,duration=2})` (speed debuff, stacks multiplicatively) Â· `freeze({duration=1.5})` (speedMul 0 + stunned â€” hard CC) Â· `poison({dps=4,duration=5,stacks})` (stacking DoT; re-apply adds a stack + retimes) Â· `stun({duration=1})` (lockout only, momentum still slides) Â· `haste({mult=1.5,duration=4})` (speed buff, cancels against slow) Â· `weaken({mult=0.5,duration=3})` (softens the entity's OWN outgoing damage via dmgMul). Node-safe and sim-testable with a fake entity and dt stepping.

## Damage resolution (reusable component â€” engine/control/damage.js)
The layer between "a hit landed" and the raw hp subtraction the health system does. A game composes MODIFIERS into one policy and asks what a hit is worth: `resolve(base, cfg, ctx) -> { amount, crit, knockback:[x,z]|null, tags, heal, pierce }`. `base` is the nominal damage (weapon/enemy stat), `cfg` is a composed policy (or an array / single modifier / null), `ctx` carries the encounter `{ from|attacker, to|target, rng }` (positions as entities or [x,y,z]/[x,z]). The verdict is PURE DATA â€” resolve never mutates entities; the caller subtracts `amount`, adds `knockback` to the target's vel, heals the attacker by `heal`, and keeps the projectile alive for `pierce-1` more hits. Modifiers: `crit({chance=0.15,mult=2})` (rng-gated multiplier â€” draws ctx.rng, no-op without it) Â· `armor({flat=0,resist=0})` (subtract flat then scale by 1-resist, floored at 0) Â· `knockback({force=6})` (unit push attacker->target * force) Â· `falloff({near=0,far=20,min=0})` (full within near, fading to min fraction at far) Â· `lifesteal({frac=0.5})` (report heal = frac * dealt; place LAST so it sees armor) Â· `pierce({count=1})` (targets one hit passes through). `compose(...mods)` folds them left-to-right (flattens arrays, skips falsy so `compose(a, cond && b)` is safe) into one policy: `ctx.damage = damage.compose(damage.crit(), damage.armor({resist:0.2}))`. Put multipliers (crit/falloff) before reducers (armor). Deterministic given ctx.rng; node-safe (no globals/timers/Math.random), so a sim seeds the rng and asserts exact numbers.

## Abilities (reusable component â€” engine/control/abilities.js)
Activatable player POWERS with cooldowns + durations â€” the discrete, button-driven counterpart to movements.js (continuous stick locomotion). Give the player ONE instance per ability (state is per-instance in the closure) and wire it to a button. Each factory returns `{ ready(), trigger(entity, dir, ctx) -> bool, tick(entity, dt) -> state }`: `ready()` asks if the cooldown is spent; `trigger(e, dir, ctx)` fires the effect only when ready (applies it, starts the cooldown, returns true; a no-op false while cooling); `tick(e, dt)` advances the cooldown + active timers each frame, runs per-frame effects, and returns `{ ready, cooldown, active, ... }` for the HUD/loop. `dir` is an XZ aim vector `[x,z]` (falls back to entity facing `rot`). Variants: `dash({distance=6,cd=1,dashTime=0.15})` (velocity burst along dir; drives e.vel while active so it overrides movement) Â· `dodgeRoll({distance=5,iframes=0.4,cd=0.8})` (dash that also opens an `e.invuln` i-frame window the health system checks) Â· `blink({range=8,cd=3})` (instant `e.pos` teleport by dir*range) Â· `shieldAbility({duration=3,cd=6})` (sets `e.shield` seconds-remaining timer) Â· `groundPound({cd=2,radius,force,damage})` (marks a `{pos,radius,force,damage}` shockwave request the game reads ONCE off `tick().shockwave`) Â· `timeSlow({factor=0.4,duration=2,cd=8})` (reports `state.timescale` 0..1 the loop multiplies into sim dt; tick with REAL dt). Pure, deterministic, node-safe. INTEGRATION: tick abilities AFTER movement.js so a dash's velocity wins for its window; effects surface as plain entity fields (`e.invuln`, `e.shield`) or tick state (`shockwave`, `timescale`) that a tiny game system / the loop applies.

## Vehicles (reusable component â€” engine/control/vehicles.js)
Set `ctx.movement` in bootstrap to drive a VEHICLE instead of on-foot locomotion â€” same `update(entity, input, dt)` shape as movements.js (`input = {move:[x,z], aim:[x,z], jump}`) but with car/boat/plane physics feel. Heading (`e.rot`) and velocity (`e.vel`) persist on the entity across frames (convention: forward = `[sin(rot),cos(rot)]`, throttle = `-move[1]`, matching movements.tank); the movement system integrates velâ†’pos as usual. Variants: `vehicles.car({accel=20,topSpeed=16,grip=6,turn=2.6})` (throttle on move[1], steering scales with speed so it can't pivot in place, lateral grip bleeds off skid so hard turns drift; low grip = slidey) Â· `vehicles.boat({accel=8,drag=1.2,turn=1.4})` (heavy momentum, omni water drag glides it to a stop, rudder bites only underway) Â· `vehicles.hover({accel=18,drag=3})` (omni-directional thrust with glide inertia â€” no fixed nose, faces its travel) Â· `vehicles.flyer({thrust=14,lift=10,turn=2})` (arcade flight: move[1] thrust along the nose, move[0] banks the heading, aim[1] climbs/dives via vertical velocity, light air drag, roll hint on `e._veh.bank` for a renderer to tilt the mesh) Â· `vehicles.heavyTank({accel=10,turn=1.2})` (pivots in place, slow inertial accel, heavy tread friction to a halt when you let off the gas). Pure and node-safe (no window/DOM/timers/Math.random); per-vehicle bank scratch lives on `e._veh`, so it sim-tests with a fake entity and dt stepping.

## Formations (reusable component â€” engine/control/formations.js)
Squad/escort/wave POSITIONING â€” the "where does each follower stand relative to the leader?" brain, swappable like behaviors/steering. Pick one per squad and ask it for a slot each frame: `const fmt = formations.wedge({ gap: 2 }); const target = fmt(i, squad.length, leader.pos, leader.rot); const v = steering.arrive(e.pos, target, e.speed); e.vel[0]=v[0]; e.vel[2]=v[2];`. Each factory returns a PURE `slot(index, count, leaderPos:[x,y,z], leaderRot) -> [x,0,z]` world target; slots are computed in the leader's local frame (local +z = forward, +x = right) then rotated by `leaderRot`, so the whole shape turns to face where the leader faces. No state, no globals â€” one instance drives the whole squad, and followers just `steering.arrive()` to their slot. Variants: `line({gap=2})` (side-by-side rank centered on the leader â€” a shield wall/firing line) Â· `column({gap=2})` (single file directly behind â€” a convoy/snake) Â· `wedge({gap=2,angleDeg=45})` (V/arrowhead with the leader at the tip, slots alternating to left/right flanks one rank deeper per pair â€” a spearhead) Â· `circle({radius=4})` (even ring around the leader, slot 0 at the front â€” a bodyguard cordon) Â· `grid({cols=3,gap=2})` (rowsĂ—columns block trailing back, columns centered â€” a phalanx) Â· `echelon({gap=2})` (diagonal stagger, each unit one step back AND to the flank â€” a slanted strafing line). Deterministic and node-safe; sim-tests by asserting slot geometry (centering, symmetry, ring radius, rotation of the whole formation).

## Pathing (reusable component â€” engine/control/pathing.js)
Route FOLLOWING and simple grid navigation â€” one rung ABOVE the steering primitives (steering.js) and a sibling of behaviors.js. Route followers return a stepper `step(entity, dt) -> [x,0,z]` (the desired velocity, also written to `e.vel` + facing `e.rot`), with per-entity progress kept on `e._path`, so ONE instance drives a whole registry. ## Utility AI (reusable component â€” engine/control/utility.js)

An optional `catalog` (plain object map `{id:def}`, a `Map`, or a `fn(id)->def`) supplies per-item `stack` overrides, so non-stackables (`{stack:1}`) each claim their own slot while the bag default covers the rest. `inventory.makeEquipment({ slots:['weapon','armor','trinket'] })` â†’ named gear slots: `equip(item, slot=item.slot)` slots an item in and RETURNS the displaced one (or null) so you can drop the old gear back into a bag Â· `unequip(slot)` returns and clears it Â· `get(slot)` reads Â· `stats()` sums every worn item's `stats` bag into one flat totals object (e.g. `{atk,def}`) Â· `all()` / `slots()` list the layout. An ITEM is plain data `{ id, stack?, stats?, slot?, ... }`; the store only tracks `{id,qty}` references and NEVER mutates the item objects you pass in. Deterministic and node-safe â€” sim-tests by adding past a stack, asserting the overflow leftover, and swapping equipped gear to check the summed stats.

## Progression (reusable component â€” engine/control/progression.js)
XP, LEVELS and unlock/skill TREES â€” the "how a player grows" meta-game system, a sibling of save/inventory. Pure data in closures (no window/DOM/timers/Math.random), so a level-up popup, a talent screen and a headless balance sim all drive the same store. `progression.makeLevels({ curve, max=99, level=1, xp=0 })` accumulates xp and ROLLS UP multiple levels when an overflow gift buys several at once: `add(amount) -> { level, leveledUp, gained }`, plus live getters `level Â· xp` (progress within the level) `Â· xpToNext` (xp remaining to next) `Â· needed` (this level's full cost) `Â· progress` (0..1 for a HUD bar) `Â· maxed Â· totalXp`, and `setLevel/reset/serialize/restore`. A `curve(level) -> xpNeeded` sets the NEXT level's cost; presets: `linearCurve(base=100)` (base*level, steady grind) Â· `expCurve(base=100,factor=1.5)` (geometric, classic RPG) Â· `table([lvl1,lvl2,...])` (hand-authored, clamps to the last entry past the end) â€” passing a plain ARRAY as `curve` is auto-treated as a table. `progression.makeSkillTree(nodes, { points=0 })` where a node is `{ id, cost?=0, requires?:[ids] }`: `unlock(id)` claims a node ONLY when every prerequisite is unlocked AND `points >= cost` (deducting the cost), returning success; `unlocked(id)/has Â· canUnlock(id) Â· met(id) Â· available()` (the frontier â€” reachable, unclaimed, affordability-agnostic so a UI can gray out too-expensive nodes) Â· `grant(n)/spend(n)` (point pool) Â· `respec()` (wipe unlocks, refund every point spent) Â· `spent Â· points` Â· serialize/restore. `progression.prestige({ levels, reward=t=>t, min=null, reset })` is the soft-reset loop: `ready()` when the watched `levels` is maxed (or hits a numeric `min`), `do()` resets the levels, bumps `tier`, banks `reward(tier)` points (feed them into a meta skill tree), plus `grant/spend/points/tier` and serialize/restore. Deterministic and node-verified; overflow multi-levels and gated unlocks are sim-testable.

## Quests / objectives (reusable component â€” engine/control/quests.js)
The GOALS side of a game â€” the objective-tracking counterpart to dialogue.js (quest flags often share the same save bag). `makeQuestLog({ onStart?, onProgress?, onComplete?, onTurnIn?, grant? })` returns a stateful log you feed EVENT NAMES (not the engine bus â€” you decide when to forward `enemy-died`/`item-picked`/`zone-entered`): `start(quest)` Â· `progress(eventName, amount=1, data?)` (advances every active objective listening for that name, returns ids of quests that just completed) Â· `completed(id)` Â· `active()` Â· `turnIn(id)` (claims the reward, gated behind completion + prerequisites) plus `get/has/all/abandon/canStart/canTurnIn/snapshot/restore`. A quest is plain data `{ id, objectives:[{ id, event, need=1, count?, optional?, hidden?, match? }], reward?, requires?, repeatable?, title? }`. An objective ticks toward `need` whenever `progress()` fires its `event` (filtered by optional `match(data)`); a COUNTER accrues over many events, a FLAG trips on the first (`need=1`). A quest completes when all REQUIRED (non-optional) objectives are met (fires `onComplete`); optional objectives track/reward but never gate, hidden ones stay out of `active()` views until revealed by progress, and finished quests keep ticking their optional/bonus goals until turned in. `requires` gates start + turn-in as an id, array of ids (all must be turned in), or a predicate over `{ turnedIn, completed, active }`; `repeatable` quests can restart after claiming (dailies). Reward dispensing goes through the optional injected `grant(reward, quest)` adapter (wallet/inventory) so the module stays pure and headless â€” no world/DOM/timers/Math.random, all state in the closure. `snapshot()`/`restore(state, catalogById)` persist objective counts across a reload (pairs with save.js). Authoring helpers: `quests.makeQuest(id, objectives, opts)`, `quests.counterObjective(id, event, need, opts)`, `quests.flagObjective(id, event, opts)`. Deterministic and sim-testable: fire event names, assert objective counts + quest completion + reward gating.

## Economy (reusable component â€” engine/control/economy.js)
CURRENCY, SHOPS, PRICING and CRAFTING â€” the "money and stuff" meta-game layer, a swappable pure-data system like inventory/save/progression (no window/DOM/localStorage/timers/Math.random; state in closures, so a HUD, a vendor screen, a loot drop and a headless sim drive the SAME numbers). `economy.makeWallet({ start=0, store?, allowDebt?, min? }) -> { balance (getter/setter), earn(n), spend(n) (debits; returns false and leaves the balance untouched when short, unless allowDebt), canAfford(n) }`; pass a `store` ({ get, set } or a signal) to keep the balance in a save file. `economy.makeShop({ stock:[{id, price, qty?, demand?}], buyMul=1, sellMul=0.5 }) -> { price(id), sellPrice(id), inStock(id), stockOf(id), buy(id, wallet?, inv?) Optional dynamic pricing: `economy.priceCurve(base, { demand=0 (-1 glut..+1 shortage), elasticity=0.5, min=1, max=Infinity })` â€” clamped, rounded; a stock row's `demand` (or `setDemand`) feeds it automatically. INVENTORY args are optional and DUCK-TYPED via `economy.invAdapter`: the engine inventory.js (add/remove/count), a plain object map `{ id: qty }`, or a Map all work interchangeably; buy/sell/craft simply route through it. Deterministic and node-safe â€” a sim asserts exact coin totals, stock counts and consumed inputs.

## Building a game from a description (the app's core job)
The app's purpose is making games. Given a description, DON'T write an engine â€” pick components and glue:
1. **View + movement** from the description: top-down shooter â†’ `cameras.topDown` + `movements.twinStick`; platformer â†’ `cameras.sideScroller` + `movements.platformer`; runner â†’ `cameras.sideScroller` + `movements.autoRun`; true 2D â†’ `cameras.flat2D`.
2. **Systems** in `game/bootstrap.js`: keep the ones the game needs (movement, ai, collision, health, score, hud, audio, render, gamestate are almost always in; add `fire`/`spawn` for shooters, drop them for a pure platformer). For a shooter also pick `ctx.weapon` (single/rapid/shotgun/burst/radial) and `ctx.aim` (stick/facing/manual/autoAim) â€” e.g. a one-thumb mobile shooter is `aim.autoAim` + `weapons.rapid`. Pick enemy/NPC brains with `ctx.behavior` (chase/orbit/zigzag/charger for foes; wander/patrol/follow/guard for NPCs), or `behaviors.byKind({...})` to mix.
3. **Entities** (`game/entities.js`): mesh (box/sphere/cylinder/plane), color, size, stats â€” pure data. **Config** (`game/config.js`): the numbers.
4. **Signals/registries** (`game/contracts.js`): add any the game needs.
5. Ask the user AT MOST 1â€“2 questions only for genuinely ambiguous core choices (e.g. "waves or endless? health or one-hit?"). Otherwise pick sensible defaults and build.
6. Only write a NEW system for behavior no component provides â€” one small file, imports only core, reacts to events/signals.

You should not need to open `engine/core`, `engine/render`, `engine/control`, or the existing `engine/systems`.
