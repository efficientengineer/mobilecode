# Engine contracts — the whole interface

This is the only engine file you normally need to read. The engine is a black
box; you build a game by writing `game/` (glue), not by editing `engine/`.

## The verbs (import from engine/core)
- `signal(v)` → `{ get(), set(v), update(fn), subscribe(fn) }` — one reactive value.
- `makeRegistry()` → `{ add(e), remove(e), each(fn), query(pred), has(e), size, flush(), clear() }` — a set of entities; add/remove are deferred (safe to mutate while iterating).
- `on(name, fn)` / `off(name, fn)` / `emit(name, data)` — the event bus. `on` returns an unsubscribe fn.
- `spawn(kind, opts)` / `spawnInto(reg, kind, opts)` / `despawn(e)` — entities. `despawn` removes from the scene AND the entity's registry.
- `makeRng(seed)` → `{ next(), range(a,b), int(a,b) }` — deterministic RNG.
- `makeLoop({ paused, registries })` → `{ start(), stop(), step(dt) }` — the frame driver.

## Entity fields
`{ kind, pos:[x,y,z], rot, scale (number or [x,y,z]), vel:[x,y,z], color:[r,g,b] (0..1), mesh, radius, dead, ...your fields }`.
`mesh` is one of: `box | sphere | cylinder | plane`. `spawn(opts.at)` sets pos.

## Phases (emitted by the loop each frame, in order)
`input → update(dt) → physics → late` (the sim; skipped when paused) then `render(dt)` (always).
Continuous state → **signals** (input sticks). Discrete happenings → **events**.

## Signals the systems expect (define these in game/contracts.js)
`gameState` (menu|playing|paused|over) · `score` · `wave` · `playerHealth` · `moveInput` [x,z] · `aimInput` [x,z] · `firing` · `player` (the entity).

## Registries the systems expect
`enemies` · `bullets`.

## Events (the vocabulary)
Raised by systems: `game-started` · `game-over` · `player-fired` · `enemy-hit {enemy,bullet}` · `enemy-died {enemy}` · `player-hit {enemy}` · `player-damaged {hp}` · `player-died` · `wave-started {n}` · `wave-cleared {n}`.

## Systems (engine/systems/*, each imports only core)
`input` (touch→sticks) · `movement` (input+velocity→position) · `ai` (runs each enemy/NPC's behavior) · `fire` (spawn bullets) · `spawn` (wave director) · `collision` (overlap→events) · `health` (hits→damage/death) · `score` · `cleanup` (life/bounds despawn) · `gamestate` (lifecycle+pause) · `hud` (DOM overlay) · `audio` (WebAudio SFX) · `render` (WebGL follow-cam).

## Cameras (reusable component — engine/render/cameras.js)
Set `ctx.camera` in bootstrap to pick a view; all are swappable without touching game logic:
`cameras.topDown({height,back})` · `cameras.sideScroller({distance,height,ahead})` · `cameras.thirdPerson({distance,height})` · `cameras.orbit({radius,height,speed})` · `cameras.fixed({eye,target})` · `cameras.flat2D({height,size})` (true 2D, orthographic top-down) · `cameras.sideScroller2D({distance,size})` (2D platformer). Each is a pure `(focus, entity, dt) → {eye,target,up?,projection?,orthoSize?}`; add your own the same way.

## Movement (reusable component — engine/control/movements.js)
Set `ctx.movement` in bootstrap: `movements.twinStick({speed})` (top-down) · `movements.eightWay({speed})` · `movements.tank({speed,turn})` · `movements.platformer({speed,jump,gravity})` (side view, gravity+jump) · `movements.autoRun({speed,jump})` (endless runner). Each is `update(entity, input, dt)` where `input = {move:[x,z], aim:[x,z], jump}`. Default is twinStick.

## Weapons (reusable component — engine/control/weapons.js)
Set `ctx.weapon` in bootstrap to pick the gun (rhythm + pattern): `weapons.single({cooldown})` (default) · `weapons.rapid({cooldown})` (machine gun) · `weapons.shotgun({pellets,spreadDeg,cooldown})` (fan) · `weapons.burst({count,gap,cooldown})` (tap-fire) · `weapons.radial({ways,cooldown})` (nova/bomb). Each is a stateful `update(dt, firing, dir) → shots[]`; a shot is `{dir:[x,z], speed?, life?, damage?, scale?}`. The fire system spawns whatever it returns, so this is the one place that decides "what a shot is." Per-shot `damage` overrides `config.bulletDamage`.

## Aim (reusable component — engine/control/aim.js)
Set `ctx.aim` in bootstrap to pick how shots are pointed: `aim.stick()` (twin-stick, default) · `aim.facing()` (shoot where you move — one-stick/auto-fire) · `aim.manual()` (sticky: flick to set, holds after release) · `aim.autoAim({range})` (lock the nearest enemy — mobile aim assist). Each is `resolve(player, raw, ctx) → [x,z]` (a unit direction). The fire system asks the aimer each shot.

## Behaviors (reusable component — engine/control/behaviors.js)
The brain for each enemy/NPC (in the `enemies` registry). Set a game-wide default `ctx.behavior`, or give one entity its own `e.behavior`, or mix types with `behaviors.byKind({kind: ..., default: ...})`. Enemy: `chase()` (default) · `flee()` · `orbit({radius,dir})` (circle-strafe) · `keepDistance({min,max})` (ranged) · `zigzag({amp,freq})` (weaves) · `charger({range,windup,dashSpeed})` (telegraph→dash). NPC: `wander({area})` (roam near spawn) · `patrol({points})` (waypoint route) · `follow({distance})` (companion) · `guard({radius})` (hold home, chase intruders). Each is `step(entity, target, dt, ctx)` and sets velocity + facing; per-entity state lives on `e._ai`, so one instance drives a whole registry. `target` is the player, or null when it's gone (NPC behaviors keep going; chasers idle).

## Spawn directors (reusable component — engine/control/spawners.js)
The "when and what to spawn" brain, swappable like cameras/movements/behaviors. Set `ctx.director` in bootstrap; the spawn system calls it each update phase as `director(dt, api)`. A director is a PURE stepper — it never imports world/rng, it only drives the injected `api = { count() (live enemy count), spawn(opts), setWave(n), rng, onWave(cb) }`, so it unit-tests with a recording fake. Variants: `waves({base,growth,delay})` (escalating arena rounds; advances to the next, bigger wave once `count()===0` and a `delay` breather passes — batch n = base+growth*(n-1)) · `endless({rate,max})` (steady time-based trickle, catches up long frames, `max` caps live count) · `burst({size,every,max})` (periodic clumps with lulls) · `boss({atWave,make,escort})` (watches the wave via `onWave`, spawns one boss + escorts once, then steps aside) · `timed({schedule:[{at,count,make}],loop})` (scripted absolute-time cues for intros/boss-rush). Compose with `chain(...directors)` — it forwards any wave announced via `setWave` into every child's `onWave`, so `chain(waves(), boss({atWave:5, make}))` lets a boss watch a wave loop. `spawn(opts)` shape is the game's own contract (opts carries { wave } | { pack } | { boss, make, escort } | { make, at }).

## Pickups (reusable component — engine/control/pickups.js)
What an item does when the player collects it — a swappable effect component. Set it per-drop on an item entity (`item.pickup = pickups.health({ amount: 50 })`), not game-wide. Each factory returns `{ kind, apply(ctx, player) }`; `apply` mutates the game's signals and/or stashes a timed buff on the player, then returns a short tag string ('heal'|'shield'|'score'|'speed'|'weapon'|'magnet'|'life') for a label/sound. Variants: `health({amount,cap})` (raise playerHealth, capped) · `shield({duration})` (player.shield invuln timer) · `scoreBonus({points})` · `speedBoost({mult,duration})` (player.speedMult+speedTimer for movement) · `weaponSwap({weapon})` (set ctx.weapon) · `magnet({radius,duration})` (player.magnetRadius+magnetTimer) · `extraLife({refill,cap})` (bump lives signal/player.lives, optional heal). Helper `makePickup(kind, apply)`. Timed buffs are seconds-remaining fields a tiny game system counts down and clears.

## Hazards (reusable component — engine/control/hazards.js)
Environmental effect zones — lava, spikes, wind, pits — swappable like behaviors. A game places a zone in the world as either a circle `{pos:[x,y,z], radius}` or an axis-aligned box `{min:[x,z], max:[x,z]}`, then picks a hazard per zone: `const lake = hazards.lava()`. Each hazard exposes `inside(entity, zone) -> bool` (radius-aware membership, works for both shapes) and `affect(entity, dt, zone) -> {damage?, push?[x,z], kill?, slow?} | null`. The verdict is PURE DATA — the hazard never mutates the entity; a system reads the fields and applies them (subtract `damage`, add `push` to vel, despawn on `kill`, scale speed by `slow` 0..1). Returns null when the entity is outside. Variants: `damageZone({dps=10})` (dt-scaled DoT) · `spikes({damage=20})` (one-shot flat hit on entry, re-arms after you leave — per-entity via WeakSet) · `lava({dps=30,slow=0.5})` (DoT + drag) · `slowField({mult=0.5})` (pure speed multiplier — mud/web) · `pit({})` (instant kill inside) · `windZone({force:[x,z]})` (constant dt-scaled velocity push — conveyor/fan/current; also exposes `outward(entity,zone)` for radial repulsors). Pure and sim-testable; a game applies verdicts wherever it steps hazards (e.g. a small system in `update`).

## Difficulty (reusable component — engine/control/difficulty.js)
Set `ctx.difficulty` in bootstrap to pick how the game ramps up. Each is a pure `scale(state) -> {speedMul, hpMul, rateMul, damageMul}` where `state = {time, score, wave, performance}`; the spawn director multiplies enemy stats by the muls when building a wave. `difficulty.flat({speed,hp,rate,damage})` (fixed, a casual baseline) · `difficulty.linear({per,cap,by,unit})` (smooth continuous ramp vs time/wave/score, default) · `difficulty.stepped({every,step})` (discrete plateaus that jump) · `difficulty.waveBased({perWave})` (keyed to the wave counter; wave 1 = base) · `difficulty.adaptive({up,down,floor})` (rubber-band/DDA: eases up when `state.performance`>0.5, down when struggling) · `difficulty.compose(...curves)` (multiply curves, e.g. a rising wave floor plus rubber-banding). Deterministic and node-safe; per-mul weights make HP/damage climb harder than speed/rate, all clamped to `cap` (default 3).

## Steering (reusable component — engine/control/steering.js)
Low-level STEERING primitives that a behavior composes (behaviors.js is the high-level brain; these are its atoms). Each returns a desired velocity vector `[x,0,z]` from pure vector math — no side effects (except `wander`, which advances a heading on the state you pass). Blend several with `combine`, write the result to `e.vel`, and let movement.js integrate it. `steering.seek(fromPos,toPos,speed)` (go there) · `steering.flee(fromPos,fromTarget,speed)` (mirror of seek) · `steering.arrive(fromPos,toPos,speed,slowRadius=3)` (ease to a stop on the goal) · `steering.pursue(self{pos,vel},target{pos,vel},speed)` (lead a moving target) · `steering.evade(self,target,speed)` (dodge the interception) · `steering.separation(selfPos,neighborsPosArray,radius,speed)` (anti-crowding push) · `steering.wander(state,{speed,jitter,turn})` (smooth roam; pass `state.rng` for determinism) · `steering.combine([{v,weight}...],maxSpeed)` (weighted blend, length-clamped). Typical use inside a behavior: `const v = steering.combine([{v:steering.seek(e.pos,t.pos,4),weight:1},{v:steering.separation(e.pos,near,2,4),weight:1.5}], 4); e.vel[0]=v[0]; e.vel[2]=v[2];`.

## Projectile motion (reusable component — engine/control/projectiles.js)
The flight path of a bullet AFTER it is fired — the complement to weapons.js (which owns when/pattern). A game runs a motion stepper over the `bullets` registry each update (a game-wide `ctx.projectile`, or `b.motion` per bullet for mixed ammo). Each factory returns `step(bullet, dt, ctx)` that mutates `bullet.vel` and/or `bullet.pos`; per-bullet state lives on `b._m`, so one instance drives the whole registry. The movement system already integrates vel→pos, so these STEER velocity and let it fly. `ctx.target(bullet) -> nearest enemy {pos}` is optional guidance (always null-guarded); `ctx.owner`/`b.owner` gives boomerang a live homing point. Variants: `straight()` (no-op, constant velocity — the default) · `homing({turn=4,speed})` (rotate toward the nearest enemy by a max turn rate, holding speed; flies straight with no target) · `boomerang({range=8,speed})` (fly out to `range` from the launch point, then curve back to the owner and self-destruct on arrival) · `wave({amp=3,freq=6})` (sine-weave perpendicular to the launch heading — snaking, hard to line up) · `arc({gravity=30})` (lob: pile downward accel onto vel[1]; launch with vel[1]>0 for grenades/mortars/arrows) · `accelerate({accel=40,max})` (ramp ground-plane speed along the heading; negative `accel` decays to a stop without reversing, `max` caps it). Pure and sim-testable with a fake bullet.

## Emitter patterns (reusable component — engine/control/patterns.js)
The TIME-driven bullet-hell brain for a hostile emitter (enemy/boss turret) — the danmaku counterpart to weapons.js (which is the player's INPUT-driven gun). A game gives an emitter its own instance and steps it each frame, spawning whatever it returns: `e.pattern = patterns.spiral({count:4}); for (const s of e.pattern(dt, {aimDir})) spawnInto(bullets, ...)`. Each factory returns a stateful `step(dt, ctx) -> shots[]` where a shot is `{dir:[x,z], speed?}` (unit XZ vector) and `ctx.aimDir` (unit vector toward the player) steers the tracking variant. Variants: `ring({count,cooldown,offset})` (evenly-spaced full-circle burst on a cooldown) · `spiral({count,turn,rate})` (rotating spokes; base angle steps `turn` radians per fire — the classic boss curl, negative `turn` reverses) · `aimedSpread({count,arcDeg,cooldown})` (a fan centered on `ctx.aimDir`, tracks the player; straight +Z with no aim) · `spinner({arms,rate,spin})` (continuous pinwheel sweeping `spin` rad/s in real time, decoupled from fire cadence) · `pulse({count,every,pulses,gap,twist})` (a heartbeat salvo of `pulses` quick twisting rings then a long `every`-second lull). All per-instance state (angles/timers) lives in the closure so each emitter needs its own instance; pure, deterministic, node-safe. Dir convention `[sin a, cos a]` matches weapons.radial and behaviors' facing math, so shots drop straight into the same fire/spawn glue.

## Targeting (reusable component — engine/control/targeting.js)
Target SELECTION strategies — the "which one do I pick?" brain shared by aim assist, homing bullets, and enemy fire. Each factory returns a selector `select(fromPos, candidates, ctx) -> entity | null`, where `fromPos` is the [x,y,z] doing the choosing, `candidates` is EITHER a plain array of entities OR a registry-like with `.each` (e.g. `ctx.registries.enemies`), and each entity has `.pos` (+ optional `.hp`). All variants skip `.dead` entities and return null on empty/null input. Pure and deterministic given `ctx.rng` (no world/DOM/timers/Math.random), so each sim-tests against a fake array of `{pos,hp}`. Variants: `targeting.nearest()` (closest — default for aim/homing) · `farthest()` (snipers/artillery) · `lowestHp()` (finish the weak — focus fire; missing hp = Infinity) · `highestHp()` (chip the tank/boss) · `random({rng})` (uniform reservoir pick via `rng` arg or `ctx.rng`; first candidate when no rng) · `inCone({dir,arcDeg,range})` (nearest inside an aim cone; `dir` XZ vector overridable per-call via `ctx.aimDir`) · `mostClustered({radius})` (target where the swarm is densest — AoE/rockets) · `leastRecent()` (round-robin across the swarm via a per-selector WeakSet that clears after a full round). Reuse ONE selector instance to preserve round-robin/rng state; selectors never mutate the candidates.

## Status effects (reusable component — engine/control/statuses.js)
Timed buffs/debuffs that ride ON an entity (in `e._status`), swappable like hazards/pickups and decoupled from their cause — a weapon, hazard zone, pickup, or behavior can all `attach` the same status. Each factory returns a per-instance status `{ kind, apply(e), tick(e,dt) -> { damage?, speedMul?, stunned?, dmgMul?, expired } }`; the verdict is PURE DATA (the status never mutates the entity — the caller subtracts `damage`, scales speed by `speedMul`, skips input while `stunned`, scales outgoing damage by `dmgMul`). Helper `attach(e, status)` runs the one-time `apply` and pushes onto `e._status`; `step(e, dt)` ticks ALL of an entity's statuses, drops the expired in place, and folds survivors into one aggregate verdict (`damage` summed · `speedMul`/`dmgMul` multiplied · `stunned` OR'd). Attach a FRESH status per victim (state lives in its closure). Variants: `burn({dps=8,duration=3})` (fire DoT, dt-scaled) · `slow({mult=0.5,duration=2})` (speed debuff, stacks multiplicatively) · `freeze({duration=1.5})` (speedMul 0 + stunned — hard CC) · `poison({dps=4,duration=5,stacks})` (stacking DoT; re-apply adds a stack + retimes) · `stun({duration=1})` (lockout only, momentum still slides) · `haste({mult=1.5,duration=4})` (speed buff, cancels against slow) · `weaken({mult=0.5,duration=3})` (softens the entity's OWN outgoing damage via dmgMul). Node-safe and sim-testable with a fake entity and dt stepping.

## Damage resolution (reusable component — engine/control/damage.js)
The layer between "a hit landed" and the raw hp subtraction the health system does. A game composes MODIFIERS into one policy and asks what a hit is worth: `resolve(base, cfg, ctx) -> { amount, crit, knockback:[x,z]|null, tags, heal, pierce }`. `base` is the nominal damage (weapon/enemy stat), `cfg` is a composed policy (or an array / single modifier / null), `ctx` carries the encounter `{ from|attacker, to|target, rng }` (positions as entities or [x,y,z]/[x,z]). The verdict is PURE DATA — resolve never mutates entities; the caller subtracts `amount`, adds `knockback` to the target's vel, heals the attacker by `heal`, and keeps the projectile alive for `pierce-1` more hits. Modifiers: `crit({chance=0.15,mult=2})` (rng-gated multiplier — draws ctx.rng, no-op without it) · `armor({flat=0,resist=0})` (subtract flat then scale by 1-resist, floored at 0) · `knockback({force=6})` (unit push attacker->target * force) · `falloff({near=0,far=20,min=0})` (full within near, fading to min fraction at far) · `lifesteal({frac=0.5})` (report heal = frac * dealt; place LAST so it sees armor) · `pierce({count=1})` (targets one hit passes through). `compose(...mods)` folds them left-to-right (flattens arrays, skips falsy so `compose(a, cond && b)` is safe) into one policy: `ctx.damage = damage.compose(damage.crit(), damage.armor({resist:0.2}))`. Put multipliers (crit/falloff) before reducers (armor). Deterministic given ctx.rng; node-safe (no globals/timers/Math.random), so a sim seeds the rng and asserts exact numbers.

## Abilities (reusable component — engine/control/abilities.js)
Activatable player POWERS with cooldowns + durations — the discrete, button-driven counterpart to movements.js (continuous stick locomotion). Give the player ONE instance per ability (state is per-instance in the closure) and wire it to a button. Each factory returns `{ ready(), trigger(entity, dir, ctx) -> bool, tick(entity, dt) -> state }`: `ready()` asks if the cooldown is spent; `trigger(e, dir, ctx)` fires the effect only when ready (applies it, starts the cooldown, returns true; a no-op false while cooling); `tick(e, dt)` advances the cooldown + active timers each frame, runs per-frame effects, and returns `{ ready, cooldown, active, ... }` for the HUD/loop. `dir` is an XZ aim vector `[x,z]` (falls back to entity facing `rot`). Variants: `dash({distance=6,cd=1,dashTime=0.15})` (velocity burst along dir; drives e.vel while active so it overrides movement) · `dodgeRoll({distance=5,iframes=0.4,cd=0.8})` (dash that also opens an `e.invuln` i-frame window the health system checks) · `blink({range=8,cd=3})` (instant `e.pos` teleport by dir*range) · `shieldAbility({duration=3,cd=6})` (sets `e.shield` seconds-remaining timer) · `groundPound({cd=2,radius,force,damage})` (marks a `{pos,radius,force,damage}` shockwave request the game reads ONCE off `tick().shockwave`) · `timeSlow({factor=0.4,duration=2,cd=8})` (reports `state.timescale` 0..1 the loop multiplies into sim dt; tick with REAL dt). Pure, deterministic, node-safe. INTEGRATION: tick abilities AFTER movement.js so a dash's velocity wins for its window; effects surface as plain entity fields (`e.invuln`, `e.shield`) or tick state (`shockwave`, `timescale`) that a tiny game system / the loop applies.

## Vehicles (reusable component — engine/control/vehicles.js)
Set `ctx.movement` in bootstrap to drive a VEHICLE instead of on-foot locomotion — same `update(entity, input, dt)` shape as movements.js (`input = {move:[x,z], aim:[x,z], jump}`) but with car/boat/plane physics feel. Heading (`e.rot`) and velocity (`e.vel`) persist on the entity across frames (convention: forward = `[sin(rot),cos(rot)]`, throttle = `-move[1]`, matching movements.tank); the movement system integrates vel→pos as usual. Variants: `vehicles.car({accel=20,topSpeed=16,grip=6,turn=2.6})` (throttle on move[1], steering scales with speed so it can't pivot in place, lateral grip bleeds off skid so hard turns drift; low grip = slidey) · `vehicles.boat({accel=8,drag=1.2,turn=1.4})` (heavy momentum, omni water drag glides it to a stop, rudder bites only underway) · `vehicles.hover({accel=18,drag=3})` (omni-directional thrust with glide inertia — no fixed nose, faces its travel) · `vehicles.flyer({thrust=14,lift=10,turn=2})` (arcade flight: move[1] thrust along the nose, move[0] banks the heading, aim[1] climbs/dives via vertical velocity, light air drag, roll hint on `e._veh.bank` for a renderer to tilt the mesh) · `vehicles.heavyTank({accel=10,turn=1.2})` (pivots in place, slow inertial accel, heavy tread friction to a halt when you let off the gas). Pure and node-safe (no window/DOM/timers/Math.random); per-vehicle bank scratch lives on `e._veh`, so it sim-tests with a fake entity and dt stepping.

## Formations (reusable component — engine/control/formations.js)
Squad/escort/wave POSITIONING — the "where does each follower stand relative to the leader?" brain, swappable like behaviors/steering. Pick one per squad and ask it for a slot each frame: `const fmt = formations.wedge({ gap: 2 }); const target = fmt(i, squad.length, leader.pos, leader.rot); const v = steering.arrive(e.pos, target, e.speed); e.vel[0]=v[0]; e.vel[2]=v[2];`. Each factory returns a PURE `slot(index, count, leaderPos:[x,y,z], leaderRot) -> [x,0,z]` world target; slots are computed in the leader's local frame (local +z = forward, +x = right) then rotated by `leaderRot`, so the whole shape turns to face where the leader faces. No state, no globals — one instance drives the whole squad, and followers just `steering.arrive()` to their slot. Variants: `line({gap=2})` (side-by-side rank centered on the leader — a shield wall/firing line) · `column({gap=2})` (single file directly behind — a convoy/snake) · `wedge({gap=2,angleDeg=45})` (V/arrowhead with the leader at the tip, slots alternating to left/right flanks one rank deeper per pair — a spearhead) · `circle({radius=4})` (even ring around the leader, slot 0 at the front — a bodyguard cordon) · `grid({cols=3,gap=2})` (rows×columns block trailing back, columns centered — a phalanx) · `echelon({gap=2})` (diagonal stagger, each unit one step back AND to the flank — a slanted strafing line). Deterministic and node-safe; sim-tests by asserting slot geometry (centering, symmetry, ring radius, rotation of the whole formation).

## Pathing (reusable component — engine/control/pathing.js)
Route FOLLOWING and simple grid navigation — one rung ABOVE the steering primitives (steering.js) and a sibling of behaviors.js. Route followers return a stepper `step(entity, dt) -> [x,0,z]` (the desired velocity, also written to `e.vel` + facing `e.rot`), with per-entity progress kept on `e._path`, so ONE instance drives a whole registry. Variants: `pathing.followPath({points:[[x,z],...], loop=true, arrive=0.4, speed})` (walk a fixed route waypoint by waypoint; advance within `arrive`, wrap when `loop` else park on the last point) · `pingPong({points, arrive, speed})` (bounce back and forth end-to-end forever; with 2 points a simple shuttle) · `patrolPath({points, wait=0, loop=true, arrive, speed})` (followPath that PAUSES `wait` seconds at each stop — sentries; `wait` is a number or a per-stop array) · `lerpTo({speed=6, arrive})` (ease toward a MOVING goal exponentially and frame-rate-independently — soft-follow companion / camera dolly; call as `step(e, goalPos, dt)`, accepts `[x,y,z]` or `[x,z]`). `gridNav({cell=1, passable(cx,cz)->bool})` is the odd one out — a PURE greedy 8-direction planner with NO world access: you hand it a `passable` predicate over integer cell coords and call `nav.stepToward(fromPos, goalPos) -> next [x,z]` (the center of the best open neighbour toward the goal, skipping blocked cells and corner-cutting diagonals, holding position when boxed in). It also exposes `toCell`/`toWorld` for aligning entities to the grid. Chain `stepToward` to crawl a route one cell at a time — cheap for swarms on mobile, no full A* alloc. All node-safe (no window/DOM/timers/Math.random) and sim-testable with a fake entity.

## Camera effects (reusable component — engine/control/camerafx.js)
Wrappers that DECORATE a base camera from cameras.js. Each is `fx(base, cfg) -> (focus, entity, dt) -> {eye,target,...}` — same shape as a raw camera, so they compose and pass `up`/`projection`/`orthoSize` straight through: `ctx.camera = cameraFx.smooth(cameraFx.shake(cameras.topDown()))`. Variants: `shake(base,{mag=0.5,decay=5,freq})` (decaying trauma jitter on the eye; fire it with the returned `.trigger(mag)` on a hit/explosion, or set `entity.shake`; seeded counter = deterministic replays) · `smooth(base,{stiffness=8})` (frame-rate-independent exponential ease of eye+target toward the base — kills snap/jitter; first frame latches) · `lookAhead(base,{lead=1.2,includeEye})` (offset target/eye by `entity.vel*lead` so the camera leads a moving subject) · `zoom(base,{min=1,max=1.6,on})` (dolly eye distance from target by a factor blended `min..max` via `on(entity)->t`, e.g. speed/health) · `deadzone(base,{radius=2,radiusZ})` (hold a lagged anchor until the focus leaves a slack box, then snap to the edge — no wobble from tiny moves). Pure, node-safe, per-instance state in closures.

## Tweening + interpolation (reusable component — engine/control/tweens.js)
The pure math of "animate a value from A to B over `duration`, shaped by an easing curve" — a value PRODUCER a game reads each frame (popup scale, fade alpha, color blend, camera dolly); it never draws or touches the DOM. `tweens.tween(from, to, duration=1, ease='linear')` returns a stepper `step(dt) -> { value, done, t }` with all progress captured in the closure (reuse ONE per animation): `value` is the eased interpolation, `t` is RAW linear progress 0..1 (assert monotonic/endpoints), `done` latches at t=1; `duration<=0` snaps instantly; `step.reset()` rewinds and `step.duration` exposes the length. A tweened value is a NUMBER or an ARRAY (lerped componentwise — positions/colors/offsets). Easing curves live under `tweens.ease`: `linear` · `easeIn` (quad accel) · `easeOut` (quad decel) · `easeInOut` · `easeOutBack` (overshoot then settle) · `easeOutElastic` (springy) · `easeOutBounce` · `smoothstep` (Hermite S-curve) — pass a curve name or your own `t->t` fn; all land exactly on 0 and 1 at the ends even when they leave the unit range mid-flight. Helpers: `tweens.lerp(a,b,t)` (number or array, extrapolates past 0..1) · `tweens.damp(cur,target,lambda=8,dt)` (frame-rate-INDEPENDENT exponential smoothing — the correct replacement for `cur+=(target-cur)*k`; number or array) · `tweens.pingPongT(t)` (fold a rising clock onto a 0->1->0 triangle wave of period 2) · `tweens.sequence([tweens])` (play tweens back to back as one animation, same `step(dt)->{value,done,t}` shape with overall `t`; time is accounted globally so long frames spill accurately into the next segment, and sequences NEST). Pure, deterministic, node-safe (no Math.random/timers/DOM/globals), sim-tested by stepping and asserting curve shape (endpoints, monotonicity, overshoot, spillover).

## Screen/game feel effects (reusable component — engine/control/effects.js)
Screen/game FEEL signals — the "juice" a renderer or the loop reads each frame. These are pure STATE MACHINES that PRODUCE values (a shake offset, a timescale, an overlay alpha, a zoom factor); they never draw or touch WebGL/DOM, and use a seeded counter instead of Math.random so replays are deterministic. Keep one instance per effect, fire it on an event, and sample it per frame. Every factory returns the same tiny shape: `{ trigger(cfg?), step(dt) -> value, active }` — `trigger` starts/restarts (optional per-fire cfg overrides), `step(dt)` advances and returns this frame's value, `active` reports whether it's still running. Variants: `screenShake({mag=0.6,duration=0.4,decay=6})` (`step -> [x,y]` decaying wobble on a seeded counter to add to the camera eye; linear timer × exponential settle; stacking takes the stronger shake) · `hitStop({duration=0.06})` (`step -> timescale` 0 while frozen else 1 — multiply into sim dt for a crunchy impact pause; tick with REAL unscaled dt; re-trigger extends) · `flash({color=[1,1,1],duration=0.2,peak=1})` (`step -> alpha 0..1` for a full-screen color overlay; carries `.color` for the renderer to tint, recolorable per trigger) · `zoomPunch({amount=0.15,duration=0.25,attack=0.25})` (`step -> zoom multiplier` around 1 that spikes then smoothstep-eases back; >1 punches in, negative `amount` punches out) · `vignettePulse({duration=0.6,peak=1,hold=0})` (`step -> alpha 0..1` edge-darkening throb that rises fast / ebbs slow, with optional `hold` seconds pinned at peak for stateful glows). Per-instance closure state, node-safe, sim-testable by triggering then stepping dt. Typical wiring: `const shake = effects.screenShake(); on('enemy-died', () => shake.trigger()); // render: const [ox,oy]=shake.step(dt); eye[0]+=ox; eye[2]+=oy;`

## Particles (reusable component — engine/control/particles.js)
CPU particle EMITTER logic — a pure DATA sim the renderer later draws (no WebGL, no DOM here; your component only PRODUCES the particle array a game hands to its point/quad drawer). A particle is `{ pos:[x,y,z], vel:[x,y,z], life, age, size, color:[r,g,b] }`. Make ONE system per effect budget: `const fx = particles.makeSystem({ gravity=-9.8, drag=0, max=2000, rng })` → `{ particles, count, emit(spec,ctx), step(dt), clear() }`. `step(dt)` ages every particle (`age+=dt`), adds gravity to `vel[1]`, applies per-second linear `drag`, integrates `pos+=vel*dt`, and drops those with `age>=life` (compacting `particles` in place — the array ref is stable). A SPEC is a stateful fn `(add, ctx) => void` that calls `add({pos?,vel?,life?,size?,color?,g?,drag?})` once per new particle (`g` overrides system gravity, e.g. sparks); `ctx = { pos, dir, dt, rand, rng }` (origin defaults to `[0,0,0]`). Preset emit-spec factories: `burst({count=16,speed=6,life=0.6,spread=Math.PI,drag=1.5})` (one-shot omni explosion pop, brakes to a hang) · `sparks({count=12,speed=7,gravity=22})` (one-shot upward-biased shards that rain back down — impacts/muzzle grit) · `fountain({rate=30,up=8})` (continuous upward jet, `rate` particles/sec, arcs down) · `stream({rate=40,dir=[0,0,1],speed=10})` (continuous directed jet; pass `ctx.dir` to swivel — thruster/flamethrower) · `trail({life=0.4,every=0})` (one near-still fading breadcrumb per call AT `ctx.pos`; `every` seconds throttles). Rate-based presets keep a fractional carry in their closure, so give each emitter its OWN instance and call it every frame with `ctx.dt`. Deterministic given `ctx.rng` (or an internal phase fallback) — no Math.random, no globals, node-safe and sim-testable (emit N, step, assert aging/expiry/counts). `particles` bundle = `{ makeSystem, burst, fountain, stream, trail, sparks }`.

## Transitions (reusable component — engine/control/transitions.js)
Screen/scene TRANSITIONS as time-based PROGRESS producers a renderer or DOM overlay reads — nothing here draws (no WebGL/DOM/timers/Math.random). Each factory returns a stateful stepper `step(dt) -> { p (0..1 eased progress), done, ...visual }`; the extra fields are exactly what the overlay paints that frame. Every stepper also carries `step.replay()` (restart), `step.reverse()` (flip direction — out<->in — and keep going), `step.reset()` (alias), and mirrors the latest `{ done, p }` onto the function itself. Variants: `fade({duration=0.5,dir="out"}) -> {alpha}` (fade TO black when dir="out", FROM black when "in") · `wipe({duration,axis="x"}) -> {p,axis}` (a hard bar sweeping across) · `slide({duration,from="left"}) -> {offset:[x,y]}` (normalized -1..1 screen displacement of a pushed panel; left/right/up/down) · `iris({duration,dir="open",center}) -> {radius 0..1}` (circular mask; "open" reveals 0->1, "close" is the cartoon iris-out 1->0) · `crossfade({duration}) -> {a,b}` (outgoing/incoming layer weights, a+b===1) · `pixelate({duration,dir="out",max=64,min=4}) -> {amount,blocks}` (chunk the frame down) · `none()` (instant cut placeholder). All accept `delay` and `easing` (from the exported `ease` set). Compose a full scene change with `pair(outT, inT, { onSwap }) -> step(dt) -> { p, phase:'out'|'in'|'done', done, out, in }`: it runs `outT` to completion, fires `onSwap` ONCE (swap scenes / tear down the old level here), then runs `inT`, reporting one blended progress (out half 0..0.5, in half 0.5..1); `reverse()` runs the whole thing backwards. Also exports `timeline` (the shared play head), `ease` (linear/quad/cubic/smoothstep/…), `lerp`, `clamp01`. Pure, deterministic, node-safe — progress is driven only by the fed `dt`, so a sim steps to completion and asserts endpoints + monotonic progress; per-instance state (elapsed, dir) lives in closures.

## Palettes (reusable component — engine/control/palettes.js)
COLOR + PALETTE helpers for the low-poly look (engine colors are `[r,g,b]`, each channel 0..1 — never 0..255). Two halves: NAMED RAMPS and HELPERS, all pure/deterministic (no Math.random/DOM/timers; variation comes from a passed rng or an index) and non-mutating (every helper returns a fresh array, so swatches pulled from a shared palette are safe to reuse). `palettes.PALETTES` is a table of 8 mood ramps (5 swatches each, ordered darkest→brightest): `sunset · neon · forest · mono · pastel · retro · ice · lava`; `palettes.palette(name)` returns a defensive COPY (falls back to mono on an unknown name). Channel math: `lerpColor(a,b,t)` (clamped per-channel blend) · `lighten(c,amt)`/`darken(c,amt)` (toward white/black) · `saturate(c,amt)` (amt -1..1, holds luma; -1 = full gray) · `grayscale(c)` (Rec.601 luma). Hue: `hueShift(c,deg)` (rotate hue via HSL, wraps at 360 any sign, holds S+L). Feedback + procedural: `damageFlash(base,t,{peak})` (hit tint — white-blown at t=0, quadratic-eases back to base by t=1, auto-clamped) · `teamColor(index,{sat,light,count})` (distinct hue per player/faction — hand-picked 8-hue wheel for small rosters, or evenly-spaced `count` hues for big lobbies; index wraps). Ramp sampling: `gradient(pal,t)` (sample a name-or-array palette as a continuous 0..1 ramp, lerp between neighbours) · `pick(pal,sel)` (one swatch; `sel` is an rng `{int|next}` for a uniform pick or a wrapping numeric index, -1 = last). Hex I/O: `toHex(c)` → `'#rrggbb'` (clamps+rounds) · `fromHex(str)` accepts `#rgb`/`#rrggbb` with or without `#` (bad input → black). A game reads results straight into `e.color` / mesh tints — e.g. `enemy.color = palettes.pick('lava', rng)`, level fog `palettes.gradient('ice', depth)`, hit flash `e.color = palettes.damageFlash(e.baseColor, e.flashT)`, player tint `palettes.teamColor(playerIndex)`. Node-safe and sim-testable (assert channel math, palette shape, hue wrap).

## Finite state machines (reusable component — engine/control/fsm.js)
The LAYERED-logic backbone that sits ABOVE behaviors.js: a behavior answers "how do I move right now?", the FSM answers "which mode am I in, and when do I switch?" (idle->alert->attack->flee->search). `fsm.makeFsm({ initial, states })` returns a reusable `step(e, dt, ctx)` that runs the current state's `update` and switches when it returns a new state name — firing the old state's `exit`, the new state's `enter`, and resetting a per-state timer. A state is `{ enter(e,ctx), update(e,dt,ctx)->nextName|null, exit(e,ctx) }` (all optional; a bare function is shorthand for `{ update }`). All per-entity state (current state name + per-state clock) lives on `e._fsm`, so ONE machine drives a whole registry. Guard helpers compose into a state's `update`: `after(seconds, toState)` (auto-transition once that many seconds elapse in the state) · `when(pred, toState)` (transition when `pred(e,dt,ctx)` is truthy — sightlines, hp, range) · `any([...legs])` (first non-null leg wins; a leg can be an ACTION fn that does work and returns null to fall through, so `any([action, guardA, guardB])` reads "act, then maybe switch"). Only ONE transition happens per step and `enter` never self-transitions, so a machine can't loop within a single frame. Read-only introspection: `is(e,name)` · `elapsed(e)` · `reset(e)` (re-arm a pooled entity). Typically wired as `e.behavior`/`ctx.behavior` (its `(e,target,dt,ctx)` shape lets each state call a behaviors.js stepper — pass the target via `ctx.target`). Pure, deterministic, node-safe (no window/DOM/WebGL/timers/Math.random; guard randomness comes off `ctx.rng`), sim-tested by driving a fake entity through dt steps and asserting `e._fsm.state`.

## Senses (reusable component — engine/control/senses.js)
PERCEPTION helpers — the layer BELOW fsm/behaviors that answers "can this enemy actually see/hear the target right now?" so foes react to what they perceive instead of omniscient world state (feeds behaviors.js / an fsm). All PURE: positions/predicates in, booleans/numbers out — no world access, no DOM/timers/Math.random. Facing convention forward=[sin(rot),cos(rot)] matches behaviors/pathing. `senses.vision({fov=90,range=12})` -> `{ canSee(selfPos, facingRot, targetPos, blocked?) }`: true iff the target is within `range` AND inside the ±fov/2 view wedge AND (if a `blocked(x,z)->bool` sampler is passed) has clear line of sight; `fov` is the FULL cone in degrees, 360 = a see-all-around sensor. `senses.hearing({radius=8})` -> `{ heard(selfPos, soundPos, loudness=1), level(selfPos, soundPos, loudness) -> 0..1 }`: an omni ear whose bubble is `radius*loudness` (gunshots carry, footsteps don't); `level` is a linear falloff (1 at the source, 0 at the edge) for graded alertness. `senses.proximity(selfPos, targetPos, range) -> bool` (cheapest sense — aggro bubble / melee reach). `senses.lineOfSight(a, b, blocked, step=0.5)` raymarches the `blocked(x,z)` predicate every `step` units (inclusive endpoints; clear when no sampler). `senses.memory({forget=3})` is the reason an enemy keeps hunting where it LAST saw you: feed `see(e, targetPos, dt)` each frame (targetPos = a live sighting to stamp a fresh fix, or null to decay the old one by dt); the fix expires after `forget` seconds. Read it back with `lastKnown(e) -> [x,y,z]|null` (the spot to search), `remembers(e) -> bool`, `freshness(e) -> 0..1` (1 = just saw it → search urgency), `visible(e)` (was the last update an actual sighting?), and `clear(e)` (target died / gave up). Per-entity state lives on e._mem, so ONE memory instance drives a whole registry. Typical wiring: `if (eye.canSee(e.pos, e.rot, player.pos, ctx.blocked)) mem.see(e, player.pos, dt); else mem.see(e, null, dt); const goal = mem.lastKnown(e);`. Node-safe and sim-testable (feed positions/predicates, step dt, assert the cone/bubble/decay).

## Group tactics (reusable component — engine/control/squad.js)
Pack COORDINATION that sits ABOVE behaviors.js: behaviors.js drives one enemy in isolation, `squad` votes on a shared plan and stamps per-entity ORDERS the behavior then obeys, turning a loose swarm into a smart, pincering unit. `squad.makeSquad(cfg) -> { join(e), leave(e), update(members, target, dt), roster }`. Enroll members on spawn (`pack.join(e)`), then each frame before the ai system call `pack.update(liveMembers, target, dt)`; it writes onto `e._squad` (PURE DATA — squad never moves the entity): `target` (the focus-fired enemy the whole pack gangs up on, or null), `role` ('flank' | 'press'), `angle` (flank bearing offset in radians, 0 for pressers), `engage` (true = attack now, false = hang back and circle — a staggered assault), `slot` ([x,0,z] world point to move to this frame), `spacing` ([x,z] anti-stack separation push), and `centroid` ([x,y,z] pack center). A behavior/steering then consumes it: `const o=e._squad; const v=steering.arrive(e.pos,o.slot,e.speed); e.vel[0]=v[0]+o.spacing[0]; e.vel[2]=v[2]+o.spacing[1];`. `update`'s `target` may be a single entity (honored as-is) or an ARRAY of candidates to vote among. cfg: `focus` ('nearest' | 'lowestHp' | 'centroid' voting strategy) · `flankers` (count >=1 or fraction 0..1 of the pack) · `spreadDeg` (flank arc) · `maxAttackers` (how many engage at once — the rest circle) · `engageRange` / `waitRange` (attacker stand-off vs. bench radius) · `spacing` / `push` (personal-space radius + strength) · `rng` (optional, only for tie-breaks). Helpers (all pure, standalone, sim-testable): `squad.centroid(members)` (living center of mass) · `squad.assignRoles(members, {flankers, spreadDeg, target})` (flank wing drawn from the FARTHEST members, alternating sides so the pack pincers; returns `{flankers, pressers}`) · `squad.focusTarget(members, candidates, strategy)` (each living member casts one ballot for the enemy IT would pick, winner = pack majority, so the swarm converges on whoever most of it can reach; ties break toward the candidate nearest the centroid). Deterministic and node-safe (no window/DOM/timers/Math.random); per-entity state lives on `e._squad`, the pack roster is a plain Set, so it sim-tests with fake entities and dt stepping.

## Dialogue (reusable component — engine/control/dialogue.js)
NPC DIALOGUE / interaction trees — the talky side of NPC logic (behaviors.js is the moving side). Author a conversation as a plain data map of nodes `{ id: { text, speaker?, effect?, next?, choices:[{ label, next, cond, effect }] } }`, then walk it with `run(tree, { start="start", vars={} })`. The runner returns `{ vars, id(), node(), choices(), choose(i), advance(), goto(id), done() }`: `node()` is the current line `{ id, speaker, text, choices }` with choices already filtered by each `cond(vars)`; `choose(i)` takes the i-th VISIBLE choice (runs its `effect(vars)`, follows `next`); `advance()` steps a LINEAR line (a node with `next` and no open choices — auto-advancing barks/cutscene text); `done()` is true at a leaf (no `next`, no visible choices) or after a choice with no `next`. Quest state is just `vars`: conds READ it to lock/unlock branches (a choice greyed out until a flag), effects WRITE it (set quest flags, grant gold) either by mutating `vars` in place or returning a patch object that's merged. `text`/`speaker`/`label`/`next` may each be a `fn(vars)` for dynamic lines and computed jumps. Pure & deterministic — all mutation lives in the passed `vars` bag (a save-game), no world/DOM/timers/Math.random — so a sim walks a branching tree and asserts gating + var mutation + endings. Terse authoring helpers on the bundle: effects `set(k,v=true)` · `inc(k,by=1)` · `toggle(k)` · `seq(...effs)`; conds `has(k)` · `missing(k)` · `eq(k,v)` · `gte(k,v)` · `all(...conds)` · `any(...conds)`.

## Utility AI (reusable component — engine/control/utility.js)
UTILITY-based decision making — the "what should I do RIGHT NOW?" brain, a sibling of fsm.js and one layer ABOVE behaviors.js: it PICKS which behavior/action to run, your glue runs it. Instead of hard states + transitions, you list candidate ACTIONS and each scores itself 0..1 from weighted CONSIDERATIONS of the world (my hp%, distance to target, ammo, threat); the reasoner picks the highest-scoring action every tick, so priorities EMERGE from the curves rather than being wired as edges. `utility.makeReasoner(actions, cfg)` where each action is `{ name, score?(e,ctx)->0..1, considerations?:[fn(e,ctx)->0..1], combine?:"mult"|"avg"|"max"|"min", weight?, run?(e,ctx) }` — supply EITHER a single `score` fn OR a list of `considerations` folded by the action's combine mode (default cfg.combine="mult", where any near-0 consideration VETOES the action, The Sims / Infinite Axis style). Returns `{ decide(e,ctx)->winning action name, act(e,ctx)->decides AND fires the winner's run(), evaluate(e,ctx)->all rows scored+sorted for HUD/debug, actions }`. `decide` is DETERMINISTIC: highest score wins, ties break by declaration order; it stashes the pick on `e._util.action`. `cfg.commit` adds hysteresis (a bonus to the held action) so the brain doesn't dither between near-ties. Consideration curves map a raw input to desirability: `linearC` (pass-through clamp) · `inverse` (mirror — near enemy / low hp reads high) · `threshold(at,soft)` (hard 0/1 gate, or smoothstep band when soft>0) · `curve(fn)` (wrap any easing, clamp to 0..1) · example curves `bell(center,width)` (sweet-spot spacing — ideal range, not too close/far) and `expo(k)` (low-end urgency). `combine(scores, mode)` folds considerations (mult=AND/veto, avg=mean, max=OR/strongest, min=weakest-link); mult scores are auto-compensated so multi-consideration actions compete fairly against single ones. Helpers `dist(a,b)` (planar XZ distance, null-safe) and `ratio(x,of)` (clamped fraction). Pure, deterministic, node-safe (no world/rng/timers/Math.random); per-entity scratch on `e._util`, so ONE reasoner drives a whole registry. Sim-tested by asserting the chosen action FLIPS as inputs change (low hp -> flee; full hp + enemy close -> attack; healthy + far -> idle).

## Save / persistence (reusable component — engine/control/save.js)
Reliable game-state save-load with NO hard-coded storage backend — a META-GAME system that runs headless. `makeSave({ store, version=1, migrate, prefix, clock })` where `store` is an injected adapter `{ get(key)->str|null, set(key,str), remove(key) }` (localStorage in-app, `memoryStore()` in tests, a file shim on a server) -> `{ save(slot,data), load(slot,fallback), has(slot), clear(slot), list(), meta(slot) }`. A slot serializes to versioned JSON `{ v, t, data }`; on `load`, if the stored `v` is older than the current `version`, it runs `migrate(oldData, fromVersion, toVersion) -> newData` to upgrade the shape, so a save shipped on v1 survives a v2 schema change. Legacy pre-versioning blobs load as v0 (then migrate), and corrupt/absent JSON returns the `fallback`. `list()` enumerates owned slots (works over `memoryStore().keys()`, a `keys` array, or a localStorage `length`/`key(i)` pair); `meta(slot) -> { v, t, stale }` powers a save-slot menu without unwrapping data. `autosave(getState, { save, slot='auto', everyMs=30000, immediate })` -> a `step(dt)` that accumulates fed dt (seconds) and persists `getState()` once the interval elapses (deterministic — no timers), plus `step.flush()` / `step.reset()`. `memoryStore(seed?)` returns a self-contained in-RAM adapter (`get/set/remove/keys/clear/size`) for tests and first runs. Pure and node-safe: never touches window/localStorage/DOM/timers/Math.random — all I/O goes through the adapter, so it sim-tests with `memoryStore`.

## Inventory + equipment (reusable component — engine/control/inventory.js)
The "what am I carrying / wearing" meta-game model — pure closure-backed DATA (no window/DOM/localStorage/Math.random), so a menu screen, a loot drop, a crafting bench and a headless sim all drive the SAME store. Two factories. `inventory.makeInventory({ slots=20, stackSize=99, catalog })` → a bag of fixed slots with stacking: `add(itemId, qty=1)` tops up existing stacks of that id, then spills into empty slots, and RETURNS the leftover that did not fit (0 = all stowed) · `remove(itemId, qty=1)` pulls across slots and returns the amount ACTUALLY removed (clamped to what's held) · `count(itemId)` / `has(itemId, qty=1)` query totals · `move(from, to)` drags one slot onto another — fills an empty target, MERGES same-id stacks up to the cap (leaving any overflow in the source), else SWAPS · `items()` snapshots the slot array aligned to indices (null = empty hole) · getters `slots` / `slotsUsed` / `free`. An optional `catalog` (plain object map `{id:def}`, a `Map`, or a `fn(id)->def`) supplies per-item `stack` overrides, so non-stackables (`{stack:1}`) each claim their own slot while the bag default covers the rest. `inventory.makeEquipment({ slots:['weapon','armor','trinket'] })` → named gear slots: `equip(item, slot=item.slot)` slots an item in and RETURNS the displaced one (or null) so you can drop the old gear back into a bag · `unequip(slot)` returns and clears it · `get(slot)` reads · `stats()` sums every worn item's `stats` bag into one flat totals object (e.g. `{atk,def}`) · `all()` / `slots()` list the layout. An ITEM is plain data `{ id, stack?, stats?, slot?, ... }`; the store only tracks `{id,qty}` references and NEVER mutates the item objects you pass in. Deterministic and node-safe — sim-tests by adding past a stack, asserting the overflow leftover, and swapping equipped gear to check the summed stats.

## Progression (reusable component — engine/control/progression.js)
XP, LEVELS and unlock/skill TREES — the "how a player grows" meta-game system, a sibling of save/inventory. Pure data in closures (no window/DOM/timers/Math.random), so a level-up popup, a talent screen and a headless balance sim all drive the same store. `progression.makeLevels({ curve, max=99, level=1, xp=0 })` accumulates xp and ROLLS UP multiple levels when an overflow gift buys several at once: `add(amount) -> { level, leveledUp, gained }`, plus live getters `level · xp` (progress within the level) `· xpToNext` (xp remaining to next) `· needed` (this level's full cost) `· progress` (0..1 for a HUD bar) `· maxed · totalXp`, and `setLevel/reset/serialize/restore`. A `curve(level) -> xpNeeded` sets the NEXT level's cost; presets: `linearCurve(base=100)` (base*level, steady grind) · `expCurve(base=100,factor=1.5)` (geometric, classic RPG) · `table([lvl1,lvl2,...])` (hand-authored, clamps to the last entry past the end) — passing a plain ARRAY as `curve` is auto-treated as a table. `progression.makeSkillTree(nodes, { points=0 })` where a node is `{ id, cost?=0, requires?:[ids] }`: `unlock(id)` claims a node ONLY when every prerequisite is unlocked AND `points >= cost` (deducting the cost), returning success; `unlocked(id)/has · canUnlock(id) · met(id) · available()` (the frontier — reachable, unclaimed, affordability-agnostic so a UI can gray out too-expensive nodes) · `grant(n)/spend(n)` (point pool) · `respec()` (wipe unlocks, refund every point spent) · `spent · points` · serialize/restore. `progression.prestige({ levels, reward=t=>t, min=null, reset })` is the soft-reset loop: `ready()` when the watched `levels` is maxed (or hits a numeric `min`), `do()` resets the levels, bumps `tier`, banks `reward(tier)` points (feed them into a meta skill tree), plus `grant/spend/points/tier` and serialize/restore. Deterministic and node-verified; overflow multi-levels and gated unlocks are sim-testable.

## Quests / objectives (reusable component — engine/control/quests.js)
The GOALS side of a game — the objective-tracking counterpart to dialogue.js (quest flags often share the same save bag). `makeQuestLog({ onStart?, onProgress?, onComplete?, onTurnIn?, grant? })` returns a stateful log you feed EVENT NAMES (not the engine bus — you decide when to forward `enemy-died`/`item-picked`/`zone-entered`): `start(quest)` · `progress(eventName, amount=1, data?)` (advances every active objective listening for that name, returns ids of quests that just completed) · `completed(id)` · `active()` · `turnIn(id)` (claims the reward, gated behind completion + prerequisites) plus `get/has/all/abandon/canStart/canTurnIn/snapshot/restore`. A quest is plain data `{ id, objectives:[{ id, event, need=1, count?, optional?, hidden?, match? }], reward?, requires?, repeatable?, title? }`. An objective ticks toward `need` whenever `progress()` fires its `event` (filtered by optional `match(data)`); a COUNTER accrues over many events, a FLAG trips on the first (`need=1`). A quest completes when all REQUIRED (non-optional) objectives are met (fires `onComplete`); optional objectives track/reward but never gate, hidden ones stay out of `active()` views until revealed by progress, and finished quests keep ticking their optional/bonus goals until turned in. `requires` gates start + turn-in as an id, array of ids (all must be turned in), or a predicate over `{ turnedIn, completed, active }`; `repeatable` quests can restart after claiming (dailies). Reward dispensing goes through the optional injected `grant(reward, quest)` adapter (wallet/inventory) so the module stays pure and headless — no world/DOM/timers/Math.random, all state in the closure. `snapshot()`/`restore(state, catalogById)` persist objective counts across a reload (pairs with save.js). Authoring helpers: `quests.makeQuest(id, objectives, opts)`, `quests.counterObjective(id, event, need, opts)`, `quests.flagObjective(id, event, opts)`. Deterministic and sim-testable: fire event names, assert objective counts + quest completion + reward gating.

## Economy (reusable component — engine/control/economy.js)
CURRENCY, SHOPS, PRICING and CRAFTING — the "money and stuff" meta-game layer, a swappable pure-data system like inventory/save/progression (no window/DOM/localStorage/timers/Math.random; state in closures, so a HUD, a vendor screen, a loot drop and a headless sim drive the SAME numbers). `economy.makeWallet({ start=0, store?, allowDebt?, min? }) -> { balance (getter/setter), earn(n), spend(n) (debits; returns false and leaves the balance untouched when short, unless allowDebt), canAfford(n) }`; pass a `store` ({ get, set } or a signal) to keep the balance in a save file. `economy.makeShop({ stock:[{id, price, qty?, demand?}], buyMul=1, sellMul=0.5 }) -> { price(id), sellPrice(id), inStock(id), stockOf(id), buy(id, wallet?, inv?) Deterministic and node-safe — a sim asserts exact coin totals, stock counts and consumed inputs.

## Touch gestures (reusable component — engine/control/gestures.js)
High-level touch GESTURE recognition — the mobile layer ABOVE the raw twin-stick input system (engine/systems/input.js turns touch into move/aim/fire; this turns touch into UI gestures for menus, maps, camera, and world interaction). Pure and headless: the game/app UI feeds it plain pointer events and a clock, and it emits flat gesture data — it NEVER touches window/DOM/WebGL/timers/Math.random, so it is deterministic and node-testable by scripting event sequences. `gestures.makeRecognizer(cfg)` returns `{ down(id,x,y,t), move(id,x,y,t), up(id,t), poll(t) -> events[], active(), reset() }` where `x,y` are screen px and `t` is a millisecond timestamp you supply. Wire the three input methods to pointerdown/move/up (each finger keyed by a distinct `id`), then drain `poll(nowMs)` once per frame and dispatch each `{ type, ... }`. Recognized gestures: `tap {x,y}` · `doubleTap {x,y}` (second tap within `doubleMs` and `doubleSlop` px of the first) · `longPress {x,y}` (a lone finger held stationary past `holdMs` — pass the clock to `poll(t)`/`move`/`up` so a motionless hold can age and fire) · `drag {x,y,dx,dy}` (emitted LIVE every move of one finger, dx/dy cumulative from the touch start) · `swipe {dir,dx,dy,vel}` and `flick {dir,dx,dy,vel}` (on release, `dir` = up|down|left|right by dominant axis, y+ = down; a flick is just a swipe past `flickMinVel`) · `pinch {scale,center}` and `rotate {angle,center}` (two fingers; `scale` is current spread ÷ the two-finger start spread, `angle` is radians turned since the pair touched, `center` is the live midpoint). Thresholds live in `cfg` with thumb-friendly defaults: `tapSlop=10` (px drift still counted as a tap/hold) · `holdMs=400` · `doubleMs=280` · `doubleSlop=30` · `swipeMinVel=0.3` px/ms · `flickMinVel=0.9` px/ms · `swipeMinDist=24` px. Release velocity comes from the last live move sample (falling back to the whole-stroke average). Multi-touch pointers are tainted so they never fire taps/swipes, and lifting one finger of a pinch re-baselines the survivor so it can't emit a stale swipe. Per-recognizer state (live pointers, queued events, double-tap memory) lives in the closure — make ONE recognizer per input surface. `gestures` bundle = `{ makeRecognizer }`.

## HUD widgets (reusable component — engine/control/hudkit.js)
HUD WIDGET logic — the pure STATE + GEOMETRY a renderer or DOM overlay draws; nothing here touches WebGL/DOM/timers/Math.random, so every widget sim-tests headless and mobile legibility lives in the DEFAULTS (fat targets, readable eases, edge-hugging blips). A game keeps one instance per widget, pokes it on events, and samples it in render. `hudkit.bar({max=100,smooth=6,value})` — a health/mana/boss/shield fill where `set(v)` snaps the true VALUE and `display` (eased by a frame-rate-independent damp toward it) chases so a hit SLIDES down instead of popping; read `.pct` for the fill to paint, `.value` for a label, `.draining` for a chip trail, `smooth=0` for rigid. `hudkit.radialCooldown({duration=1,charges=1})` — an ability ring: `trigger()` spends a charge only when able (returns did-fire), `step(dt)` refills, `.fraction` 0..1 is the ring (1=ready), `.ready`/`.remaining` for the button. `hudkit.toastQueue({max=3,life=2,fade=0.25})` — a self-ageing pickup/notification stack: `push(msg,opts)` drops on top (shoves the oldest past `max`), `step(dt)` expires by age, `items()` returns render rows carrying `t` 0..1 and `alpha` (fades over the final `fade` fraction). `hudkit.comboCounter({window=2})` — a decaying hit chain: `hit(n)` bumps + re-arms the window, `step(dt)` bleeds it and DROPS `count` to 0 when it lapses; `.broken` latches true for the single expiry step (flash "BREAK"), `.fraction` is the window pip, `.best` tracks the session. `hudkit.minimap({worldSize,mapSize,center})` — world->map projection (`worldSize` full extent scalar or [w,h], `mapSize` pixels, `center` [x,z] you can re-set each frame for a scrolling radar): `project(worldPos)->[x,y]` in map space (y-down), `clampToEdge(mapPos)` pins an offscreen blip to the border along its bearing so enemies never vanish, `blip(pos)->{x,y,off}` is the one-call render path. `hudkit.timer({from=60,dir='down',cap})` — a counting clock: `step(dt)` advances only while running, `dir:'down'` counts to 0 and latches `.done`, `'up'` counts (to an optional `cap`); `.label` is zero-padded "mm:ss", `.seconds` the raw number, `add(sec)` for +time pickups. Deterministic, node-safe, per-instance closure state; sim-tested by stepping dt and asserting the ease, expiry, reset, projection, and clock.

## Menus (reusable component — engine/control/menus.js)
MENU / navigation STATE — the model behind title screens, pause menus, and settings panels; the renderer DRAWS while this holds the cursor + values and turns input intents into plain data (pure, node-safe, state in closures — no window/DOM/timers/Math.random). `menus.makeMenu(items)` drives one screen where an item is `{ id, label, type:"button"|"toggle"|"slider"|"choice", value?, min?, max?, step?, options?, disabled? }`: button = a command, toggle = a boolean, slider = a number clamped to `[min,max]` stepped by `step`, choice = a labelled pick from `options`. Returns `{ move(dir), adjust(dir), activate(), focused(), value(id?), values(), setCursor(id), setDisabled(id,on), find(id), items, cursor, length }`. `move(dir)` walks the cursor up/down and WRAPS, skipping `disabled` rows so focus never lands on a greyed option (mobile-friendly — no thumb dead-ends). `adjust(dir)` edits the focused row: left/right flips a toggle, nudges a slider by `step` (re-clamped/quantized to a legal notch), or cycles a choice (wraps); returns the new value. `activate()` confirms the focused row — a button returns its `id` (the fire signal for your glue), a toggle flips and returns its `id`, slider/choice return null (no confirm action). Introspect with `focused()` (the row), `value(id?)` (focused or named value; a choice resolves to its option label), and `values()` (a flat `{id:value}` snapshot for saving/feeding config, buttons omitted). Sliders quantize their initial value to the nearest notch. `menus.makeScreens(initial)` is the screen STACK behind nested menus + a hardware back button: `{ push(name), pop(), replace(name), reset(name?), top(), has(name), depth, stack }` — `push` opens a submenu, `pop` is the back button (drops the top, returns the revealed screen, refuses to empty past the root), `replace` swaps the visible screen without deepening. Deterministic and sim-tested (cursor wraps + skips disabled, slider clamps/steps, toggle flips, choice cycles, stack push/pop/replace/depth).

On-screen TOUCH CONTROL placement for phones — the mobile-ergonomics layer that answers "where do the virtual joystick, action buttons and HUD labels physically sit on THIS screen?" so nothing hides under the notch or falls outside the thumb's reach. ## Tutorial / onboarding (reusable component — engine/control/tutorial.js)
Drives a first-run WALKTHROUGH step by step so the UI just paints `current()` and forwards events — the onboarding counterpart to quests.js (goals) and dialogue.js (talk). `tutorial.makeTutorial(steps, { flag?, onStep?, onDone?, state? })` where a step is plain data `{ id, text, target? (a HUD/world anchor the UI spotlights — any token, we never read it), advanceOn? (event NAME or array that clears the step), until? ((state,dt)->bool state-driven gate), gate? (block OTHER input while up: true = all, ['fire'] = whitelist the verb being taught, or (name)->bool), optional?, skipIf? (state->bool — the optional step drops itself the moment its condition already holds) }`. ## Tilemap (reusable component — engine/control/tilemap.js)
The GRID foundation for rooms/dungeons/overworlds (Zelda-style) — pure tile DATA laid on the ground plane (XZ, y up), the thing gridNav (pathing.js) navigates and a renderer draws. A tile is a plain value: a number (a type id) or an object of flags `{ solid, water, hazard, type, ... }`. `tilemap.makeTilemap({ w=8, h=8, fill=0, tileSize=1 }) -> { get(tx,tz), set(tx,tz,v), inBounds(tx,tz), worldToTile([x,z]|[x,y,z])->[tx,tz] (floor), tileToWorld(tx,tz)->[x,0,z] (cell CENTER at (i+0.5)*tileSize), tileAtWorld(pos)->{tx,tz,v}, flag(tx,tz,name), isSolid(tx,tz), passable(tx,tz), forEach(cb(v,tx,tz)), layer(name)->{get,set,forEach}, w, h, tileSize, cells }`. `isSolid` reads a tile's `.solid` flag and treats OUT-OF-BOUNDS as solid, so the world is walled by default and nothing paths off the edge; `passable = (tx,tz)=>!isSolid` is exactly the predicate `pathing.gridNav({ passable })` consumes (coordinates line up — same floor/center convention — so hand it straight over; don't import pathing). Authoring: `tilemap.fromStrings(rows, legend, { tileSize, fill })` builds a room from ASCII ART — `rows` is an array of strings (rows[0] = tz 0, the top row), width = the longest row, each char looked up in `legend` (char -> value/object; object values stored BY REFERENCE so every wall shares one flag object); unknown chars fall back to `fill`. `tilemap.toStrings(map, legend?)` (also `map.toStrings(legend?)`) renders back to strings for round-trip/debug, matching a legend char by reference then value-equality. Optional named LAYERS (ground/decor/collision) via `layer(name)` share the grid's dimensions with their own cells. Pure and node-safe (no window/DOM/WebGL/timers/Math.random); sim-tests by authoring from strings and asserting coordinate maps, bounds and solidity. Typical wiring: `const room = tilemap.fromStrings(art, legend); const nav = pathing.gridNav({ cell: room.tileSize, passable: (cx,cz)=>room.passable(cx,cz) });`

## Tile collision (reusable component — engine/control/tilecollision.js)
WORLD-layer collision the engine lacked: it resolves an entity moving against SOLID TILES (the built-in collision system is entity-vs-entity only), giving Zelda-style wall-sliding room navigation. The game supplies `isSolid(tx,tz)->bool` over INTEGER tile coords (e.g. a tilemap.isSolid) — the SAME predicate `pathing.gridNav` consumes, so pathing and collision agree on what a wall is; this file never reads the world. Ground plane XZ, y untouched. Core: `tilecollision.moveAndCollide(entity, dx, dz, isSolid, { size=0.4, tileSize=1, cornerCut=0 }) -> { moved:[mx,mz], hitX, hitZ }` — moves the body's AABB (half-extent `size`) by (dx,dz) but SEPARATES THE AXES: try X and snap flush against the first solid column hit, then try Z and snap flush, so a diagonal into a wall cancels the blocked axis and keeps sliding on the free one; `moved` is the displacement ACTUALLY applied per axis (0 = stopped), `hitX/hitZ` flag wall contact, and it writes `entity.pos`. Set `cornerCut` > 0 (try ~0.4*size) to ROUND convex corners — when a mostly-single-axis push is blocked it nudges perpendicular (up to cornerCut) to slip the player into a doorway they clipped. Extras (all pure, take positions/predicates): `resolveCircleVsTiles(pos, radius, isSolid, tileSize=1) -> { pos:[x,y,z], push:[px,pz], hit }` (DEPENETRATE a circle already overlapping solids — push out along the deepest face, iterated so a corner-wedged body pops free; for spawns/teleports/knockback, never mutates the input) · `rayTile(from, to, isSolid, tileSize=1) -> { tile:[tx,tz], point:[x,y,z], dist, t } | null` (first solid tile a ray CROSSES via grid DDA — the NEAR wall, for line-of-sight / aim laser / hitscan; a ray starting inside a solid reports it at dist 0, a clear segment returns null) · helpers `tileOf(x,z,tileSize)` and `overlapsSolid(x,z,r,isSolid,tileSize)`. Deterministic and node-safe (no window/DOM/timers/Math.random), sim-tested by walking a body into a wall (stops one axis, slides the other), depenetrating an overlap, and raycasting to the near wall. Integration: run it in the movement phase from moveInput, e.g. `const step = movement.speed*dt; tilecollision.moveAndCollide(player, moveInput[0]*step, moveInput[1]*step, tilemap.isSolid, { size:player.radius, tileSize, cornerCut:0.15 });` — replace the raw `pos+=vel*dt` integration for tile-bound entities; feed `tilemap.isSolid` to both this and `pathing.gridNav({ passable: tilemap.isSolid })` so enemies path and the player collides against one wall definition.

## Rooms / screen graph (reusable component — engine/control/rooms.js)
ROOM / SCREEN graph with doors — the structural spine of a Zelda-style world (screen-by-screen overworld, or a dungeon of connected rooms). Where tilemap.js owns ONE room's grid, rooms.js owns the GRAPH: which rooms exist, which door leads where, which are locked, where the camera is boxed in, and what's been explored. `rooms.makeWorld({ rooms, start, doorMargin=0.75, doors? })` builds a PURE state machine over plain room/door data — no world/DOM/tilemap import. A room is `{ id, bounds?:{min:[x,z],max:[x,z]}, tilemap?, spawns?, neighbors? }`; `bounds` is a ground-plane (XZ) box and auto-derives from `room.tilemap` (`w*tileSize` × `h*tileSize`) when omitted. `world.door(fromId, edge|pos, toId, { locked, key, at, span, radius, id })` places a DIRECTIONAL door in the FROM room — either an EDGE band (`'north'|'south'|'east'|'west'` of the room bounds, screen-oriented like tilemap ASCII: north=top=min z, optionally clipped to a `span:[lo,hi]` on the perpendicular axis) or a circular `{at:[x,z],radius}` spot; `room.neighbors:{east:'B',...}` auto-wires implicit edge doors (list both sides for a two-way passage). API: `current()`/`currentId()` (active room) · `enter(roomId,{at})` (INSTANT screen-cut: switch active room, mark visited, record the landing spot, return the room with `.entryAt` set) · `tryDoor(playerPos)` (REPORTS without mutating whether the player is over a door of the current room → `{ to, locked, needs, at, door }`; a locked door's `needs` is the key it wants — or `true` for a switch — so the game consumes one from inventory; `at` is the landing spot to pass straight to `enter()`, computed just inside the destination's opposite edge preserving the crossing coordinate) · `unlock(idOrKey)` (open a door by id, or SPEND a key to open every door needing it now and later) · `lock(id)` (re-lock a trap/puzzle door) · `isLocked(id)` · `doorsOf(roomId)` · `cameraBounds()` (the current room's `{min,max}` box for the follow-cam to clamp — the boxed-in screen feel; null for an open room) · `visited(id)`/`markVisited(id)`/`visitedRooms()` (the dungeon map). Transitions: switching is instant by default; for a fade/slide DON'T call `enter()` immediately — run a transitions.js stepper and call `enter()` at its midpoint. `world.isSolid` equivalent lives on the room's tilemap, which feeds pathing.gridNav's `passable` — coordinate conceptually, don't import. Node-safe (no window/DOM/WebGL/timers/Math.random), deterministic, sim-tested by walking a position through a door and asserting the active room changed and a locked door blocks until unlocked.

## Interaction (reusable component — engine/control/interact.js)
The Zelda overworld A-button staple: walk up to a chest/sign/NPC/lever, see a button hint, press A. `interact.makeInteractions({ range=1.5, needFacing=true, facingDot=0.35 })` returns a registry `{ add, remove, all, size, nearest, prompt, trigger, hold, releaseHold, clear }`. `add(x)` takes EITHER a live entity (it reads `pos/prompt/onInteract/facing/once/hold` off the entity itself) OR a plain spec, and returns a handle; the entity is held by reference so a moving NPC's `pos` is re-read each query. `nearest(playerPos:[x,y,z], playerFacing:[x,z], { range, needFacing, facingDot })` returns the CLOSEST interactable within range that is in FRONT of the player — `dot(playerFacing, dirToTarget) >= facingDot` (so you grab what you look at, not what's behind you) — and, if the item carries its own `facing`, only when approached from its front side (a sign you read from the front); spent one-shots are skipped so their hint disappears. `prompt()` is the current nearest's hint text (a string or a `fn(of)`), for a button-hint UI. `trigger(player, ctx)` re-resolves the nearest from the passed player (deriving facing from `player.facing` or `player.rot`→`[sin,cos]`) and fires its `onInteract(of, player, ctx)`, honoring `once` (a chest opens exactly once; a lever re-fires forever), returning `{ interactable, result }` or null. `hold(player, dt, ctx)` is hold-to-interact: it accumulates time on the nearest and fires once at `it.hold` seconds, returning `{ progress 0..1, fired, interactable }`, resetting when the target changes or is lost; items with no `hold` fire instantly (tap), and `releaseHold()` cancels a partial fill. Per-item `radius`/`facing`/`once`/`hold` override the registry defaults. Its `nearest` doubles as the passable/interact probe that pairs with pathing.js and tilemap solidity conceptually. Pure and node-safe (no world/DOM/timers/Math.random) — positions and facing are inputs — so it sim-tests with fake data (nearest picks the closest in-front target; behind/out-of-range excluded; once fires a single time; prompt reflects the target).

## Character stats (reusable component — engine/control/stats.js)
The number spine of a turn-based JRPG (FF6-style) — a sibling of the field-layer progression/inventory systems but built for the BATTLE half, and deliberately independent of the real-time damage.js/statuses.js. PURE DATA: a unit's stat block, live HP/MP pool, growth curve and equipment overlay live in a closure (no engine deps, no window/DOM/timers/Math.random), so a battle scene, a status menu and a headless balance sim all drive the SAME store. `stats.makeStats(base, opts)` wraps `{ level, hp, mp, vigor(str), speed, stamina, magic, defense, mdefense, evade, mblock }` — hp/mp are VITALITY SEEDS (the level-1 pool baseline), the rest are combat stats. It DERIVES `maxHp` (hp seed + stamina, scaled by level) and `maxMp` (mp seed + magic), both strictly increasing in level, and keeps current hp/mp clamped to [0,max]. API: `get(name)` (effective stat folding growth+gear; also serves level/hp/mp/maxHp/maxMp) · `set(name,val)` (overwrite a base seed, recompute caps) · `hp`/`mp`/`maxHp`/`maxMp`/`level` getters · `damage(n)->newHp` (clamps to 0 = KO) · `heal(n)` (clamps to max, no-op while KO'd) · `spendMp(n)->bool` (false when short, pool untouched) · `restoreMp(n)` · `isKO` (hp<=0) · `alive` · `revive(frac=0.5)` (raise a KO'd unit to frac of maxHp, min 1) · `full()` (inn/rest) · `snapshot()` (save blob). GROWTH: `levelUp()` / `setLevel(n)` re-derive the block from the base seed at the new level (clamped 1..99), preserving damage already taken (grows the cap, not the wound); pass `opts.growth` as either a curve fn `(baseBlock, level) -> grownBlock` or a partial per-stat increment table (defaults cover every key). EQUIPMENT: `equipmentMods(bag)` folds flat +stat bonuses as a REPLACEABLE overlay on top of growth — accepts a plain `{vigor:+5,...}` object or anything with a `.stats()` method (round-6 equipment), and `equipmentMods(null)` unequips. Deterministic and node-safe; sim-tested by asserting damage/heal clamps, KO at 0, failed MP spend, and level/gear raising maxHp.

## Elemental affinity (reusable component — engine/control/elements.js)
The pure ELEMENTAL AFFINITY table for the turn/ATB battle core (FF6-style) — the lookup that turns "a fire spell hit this defender" into a signed number, decoupled from the actual hp math (damage.js / a battle resolver owns that). A defender carries an affinity map `e.affinity = { fire:"weak", ice:"resist", water:"null", poison:"absorb" }`; a spell/attack carries one or more element strings. Verdicts: `weak` -> x2 (double), `resist` -> x0.5 (half), `null` -> x0 (immune), `absorb` -> x-1 (HEALED — the returned amount goes negative), unlisted/`none` -> x1. Core: `elements.multiplier(element, affinity) -> number` (affinity is a defender map OR a bare verdict string; element null = non-elemental, always x1) · `elements.resolve(element, defenderAffinity, amount, opts?) -> { amount (SIGNED — negative = healed via absorb), tag ('weak'|'resist'|'null'|'absorb'|'normal'), mult, element }` (rounds toward zero unless `opts.round===false`). Helpers: `ELEMENTS` (the 8 canonical element constants — the table is open, add your own) · `AFFINITY` (verdict->multiplier map) · `makeAffinity(overrides)` (fresh map, every element defaults 'none', spread overrides on top — a defensive copy so shared enemy templates never alias) · `isImmune(element, affinity)` (x0) · `absorbs(element, affinity)` (heals) · `combine(a, b, mode)` folds two verdicts/multipliers for a multi-element attack (`'best'` attacker-favouring picks the weakness, `'worst'` defender-favouring lets any null/absorb dominate, `'product'` compounds; returns a named verdict when it lands on one, else the raw number) · `resolveMulti(elements[], defenderAffinity, amount, {mode})` runs a whole element list through combine then applies it. Pure and deterministic — NO rng (affinity is a lookup, not a roll), no world/DOM/timers, never mutates the defender map — so a sim asserts exact multipliers (weak doubles, resist halves, null zeroes, absorb flips to healing).

## Spells + casting (reusable component — engine/control/spells.js)
Turn-based SPELL/ABILITY data + a cast resolver — the JRPG (FF6/ATB) counterpart to weapons.js, and a standalone system SEPARATE from the real-time damage.js/statuses.js. A spell is plain data: `{ name, mp, power, element?, kind, target, status?, statusChance?, hits=1 }` where `kind` is `attack|magic|heal|revive|drain|buff|debuff|status` and `target` is `one-enemy|all-enemies|one-ally|all-allies|self|ally-row|random-enemy`. `spells.makeSpellbook(defs)` indexes an array-or-map of defs into `{ get(name), has(name), all() }`. `spells.targets(spell, { allies, enemies, chosen }, ctx)` resolves the scope to the concrete unit list (`chosen` = the player's picked unit for one-*/ally-row; `ctx.caster` fills `self`; `random-enemy` uses `ctx.rng`). `spells.cast(caster, spell, targetsList, ctx) -> results[]` spends `caster.mp` (and BLOCKS the entire cast, spending nothing, when short — the returned array carries `.cast === false`), then for each target computes the effect via an INJECTED resolver `ctx.resolve(caster, spell, target, ctx)` (falling back to the built-in `spells.defaultResolve`: physical atk/def or magical mag/mdef minus defense, times an element multiplier from the target's `weak|resist|immune|absorb` lists, with rng-gated miss/crit), so spells.js NEVER hard-depends on a formula module. `cast` mutates in place like a turn engine: heals cap at `maxHp` and skip KO'd allies, `revive` affects ONLY the fallen (restores `power<=1` as a fraction of maxHp), `drain` damages the foe and heals the caster, element `absorb` flips damage to healing, `buff/debuff/status` apply only a rider, and `status`/`statusChance` push a status NAME onto `target.statuses` for a battle system to realize. Each result is `{ target, amount, healed, crit, missed, status }` (+ `revived`/`drained` where relevant). `spells.canCast(caster, spell)` is the MP predicate. Sample spells `Fire/Firaga/Bolt/Ice/Poison/Cure/Cura/Drain/Life` ship in `spells.SPELLS`. Pure, deterministic (inject `ctx.rng` as a `()=>0..1` fn or a core `makeRng()`), node-safe and sim-testable on plain stat objects.

## JRPG battle formulas (reusable component — engine/control/formulas.js)
The number kernel of a TURN-BASED / ATB battle command (FF6-flavored) — the turn-based sibling of the real-time `damage.js`. Pure functions take attacker/defender STAT BLOCKS (a `stats.js` store read via `.get(name)`, OR a plain `{vigor,magic,defense,mdefense,evade,mblock,level}` object), an options bag, and an INJECTED rng for variance; whole, sign-carrying numbers come out (a NEGATIVE amount = the hit was absorbed and HEALS the target). No window/DOM/timers/Math.random, so a headless balance sim gets the exact same numbers as the battle scene. `physical({atk,def,power=1,level=1,variance=0.15,crit,rng}) -> {amount,crit}` (effective attack = vigor*2 + weapon, times weapon `power` and a level term, mitigated by FF6's `(256-def)/256` curve — every defense point shaves ~0.4%; `crit:true` -> x2 or pass a number) · `magical({magic,mdef,power,level,variance,rng}) -> {amount}` (spell-`power`-dominant, boosted by caster magic+level, mitigated by mdef; no crit) · `heal({magic,power,level,variance,rng}) -> amount` (cure math, defense-free, positive HP) · `hitChance({accuracy=1,evade=0}) -> 0..1` (inputs are fractions OR percentages >1 auto-/100; `acc*(1-evade)`) · `rollHit(rng,chance) -> bool` (auto-hits when chance>=1 or no rng) · `critRoll(rng,chance=0.25) -> bool` · `variance(rng,amount,spread=0.15)` (±spread jitter, stays within bounds) · `backRow(amount)` (halve melee). The compose helper `resolveAttack({attacker,defender,weapon|spell,rng,element,elements,backRow,resolveElement}) -> {amount,crit,missed,element,tag}` runs the FF6 pipeline in order HIT (hitChance+rollHit; a miss short-circuits, physical reads defender `evade`, spells read `mblock`) -> BASE (physical for a weapon, magical for a spell) -> ELEMENT (built-in weak/resist/null/absorb table on `defender.affinity`, or inject `opts.resolveElement` to delegate to elements.js) -> CRIT (physical only) -> VARIANCE, then rounds; `tag` labels the element result ('weak'|'resist'|'null'|'absorb'|'normal'|'miss') for a floater/log. Deterministic given the rng seed; sim-testable (higher def lowers damage, crit multiplies, high evade misses, variance in-bounds, absorb goes negative).

## ATB battle core (reusable component — engine/control/battle.js)
The turn/time-based heart of an FF6-style fight — the one TURN-BASED paradigm in an otherwise real-time engine (do NOT confuse with the action-game damage.js/statuses.js). `battle.makeBattle({ allies, enemies, rng, atbSpeed=1, perform, wait=false, threshold=1 })` runs an ACTIVE TIME BATTLE: each unit (a stats.js block with at least `{ id, side:"ally"|"enemy", speed, alive }`) carries an ATB gauge that fills by `speed*dt*atbSpeed*rate` to `threshold`; a full gauge makes the unit READY and it must be given an action. battle owns TIMING/ORDER/flow ONLY — it never does damage math; it calls the INJECTED `perform(action, ctx)` callback (which mutates units/stats and returns results) and just watches `unit.alive`/`hp`/`isKO` to settle deaths and wipes. API: `step(dt) -> events[]` (fills gauges, then surfaces/resolves in one pass, returning `gaugeFull{unit}` · `acted{unit,action,results}` · `ko{unit}` · `victory` · `defeat` · `fled`); queued actions resolve oldest-ready-first (whoever filled earliest acts first), the acting unit's gauge resets to 0, and a fully-downed side ends it (enemies down = victory, party down = defeat). `submit(unitId, action)` queues a READY unit's chosen move (false if not ready/dead); `ready()` lists units awaiting a command; `order()`/`next()` give predicted turn order (soonest-to-ready first, ready = 0s); `gauge(id)`/`isReady(id)`. Haste/slow via a per-unit gauge-rate multiplier: `setRate(id,mult)` · `haste(id,mult=2)` · `slow(id,mult=0.5)` · `rate(id)`. `pause()`/`resume()` freeze gauges while a menu is open, and `wait:true` (FF6 Wait mode) auto-freezes all gauges whenever a unit is ready and awaiting a command. `escape(rng=,chance=0.5)` rolls `rng.next()<chance` to flee (ends the fight; a `fled` event surfaces on the next `step`). `isOver()`/`result()` ('victory'|'defeat'|'fled'|null). The `ctx` handed to `perform` = `{ actor, rng, battle, allies, enemies, units, foes, party, foesOf, alliesOf, unit(id) }` with `foes`/`party` pre-filtered to LIVING units so damage code never targets a corpse. Pure, deterministic, dt-driven, rng injected — no Math.random/timers/DOM; all per-unit ATB state lives in a closure Map, never module globals; sim-tests by stepping fixed dt with a fake `perform` and rng.

### party — roster / active-party management (control/party.js)
`party.makeParty({ roster=[], activeMax=4 })` manages a JRPG cast where a member is plain data `{ id, name, stats (a stats.js store), equipment?, spells? (learned), row:"front"|"back" }`. It wraps round-9 stats blocks — it never does damage/ATB math. Returns:
- `roster()` — all members (copy, insertion order) · `byId(id)` — lookup or null.
- `active()` — the up-to-`activeMax` fielded members, leader first · `bench()` — everyone not active · `leader()` — `active()[0]`.
- `add(member)` — recruit (fills an empty active slot if under cap) · `remove(id)` — drop entirely (also from active).
- `setActive(ids)` — choose the fielded party: keeps order, skips unknowns, dedupes, caps at `activeMax` (leader = ids[0]).
- `swap(idA,idB)` — both active → reorder; one active + one benched → bencher takes the active slot; else no-op. Returns whether it moved.
- `setRow(id,row)` / `rowOf(id)` — front/back stance (back row halves physical dmg; formulas.js reads `member.row`); non-"back" normalizes to "front".
- `inActive(id)` · `alive()` (active members not KO) · `wiped()` (all active KO).
- `formationSlots(side="ally")` — maps the active party to BATTLE SLOTS ready for `makeBattle({ allies|enemies })`: each slot `{ id, name, side, slot, row, member, stats, equipment, spells }` plus live getters `speed/hp/alive/isKO` delegating to the wrapped stats store, so ATB timing and KO checks stay canonical.
Pure data, deterministic, no engine deps / DOM / timers / Math.random; all state in the closure. `export const party = { makeParty }`.

## Turn/ATB ailments (reusable component — engine/control/ailments.js)
The battle-side STATUS AILMENTS of an FF6-style fight — the turn/ATB counterpart to the real-time statuses.js (do NOT reuse that one here; this speaks the ATB's language, not speedMul/stunned). Each ailment is DATA+BEHAVIOR built by `makeAilment(kind, cfg) -> { kind, gaugeRate (multiply ATB fill: haste 2 / slow 0.5 / stop 0), reflect, out, duration/remaining/expired, curedBy:[tags], onTick(unit,dt)->{damage?} (DoT/heal over time), onReady(unit)->{skip?,forceAction?,blockMagic?,out?} (consulted when a unit becomes ready) }`. Imports NOTHING — units are plain stat blocks (a stats.js store or any `{maxHp, damage()}`), no rng (ailments are lookups/timers, deterministic). Presets: `poison({dps=12})` / `sap({frac=0.03})` (flat vs %-of-maxHp DoT) · `regen({hps=10})` (heal over time = negative damage) · `sleep()` (skip until a hit wakes it) · `muddle()`/`confuse` (forceAction a RANDOM target) · `berserk()` (force-attack a foe + block magic) · `silence()`/`mute` (blockMagic) · `haste({mult=2})` · `slow({mult=0.5})` · `stop({duration=6})` (gaugeRate 0 lockout) · `petrify()`/`stone` (gaugeRate 0 + `out` until cured) · `reflect()` (bounce-magic flag). Helpers on `unit._ail`: `attach(unit, ailment)` (no dup of a kind — refreshes duration) · `step(unit, dt) -> { damage (summed, negative=net heal), gaugeRate (PRODUCT), skip, blockMagic, reflect, out, forceAction, expired:[kinds] }` (ticks DoT, ages durations, drops expired) · `ready(unit)` (folds only the on-ready hooks, no time passes — call exactly when the gauge fills) · `gaugeRate(unit)` (rate product shortcut) · `cure(unit, kindOrTag)` (by kind 'poison' or curedBy tag 'remedy'/'esuna'/'dispel'/'goldneedle') · `wake(unit)` (clear everything curedBy 'hit' — sleep/confuse) · `has/get/list/clear`. Wiring: `const a = step(u, dt); battle.setRate(u.id, a.gaugeRate); if (a.damage) u.damage(Math.round(a.damage));` and on gaugeFull consult `ready(u)` for skip / forced command / blocked magic; `wake(target)` on a landed physical hit. Pure, deterministic, node-safe (no window/DOM/timers/Math.random; state in the ailment objects + unit._ail, never module globals); sim-tested (poison ticks, stop zeroes the gauge, sleep skips then a hit wakes, silence blocks magic, durations expire and drop).

## Random encounters (reusable component — engine/control/encounters.js)
The field-exploration bridge INTO the ATB battle core (battle.js) — the turn-based cousin of the real-time spawn directors (spawners.js): as the party walks a zone this fills a THREAT meter and trips it into a random battle, weight-picking an enemy FORMATION and rolling the FF6 first-strike flags. `encounters.makeEncounters({ rate=0.08, tables, rng, danger=1, grace=3, preemptive=0.06, backAttack=0.06, cap=1 })` where `tables` maps a `zoneId -> [{ weight, formation, minLevel? }, ...]` and `rng` is an injected `() => 0..1` fn (the default; a core `makeRng()` `{next}` also works). `step(zoneId, moved=1, { level? }) -> { formation, zone, preemptive, backAttack } | null` accumulates `rate*danger*moved` into a threat meter each field step (moved = distance or 1 per tile) and rolls the meter — a hit returns the encounter and clears the meter, else null; the meter climbs to a `cap` PITY ceiling so a dry streak can't last forever, and a post-battle GRACE distance (armed by `reset()`) is bled off first so the very next step can't instantly re-ambush. `pick(zoneId, level)` weight-selects a formation, filtering out entries whose `minLevel` exceeds `level` (falling back to the whole list so a zone always yields one); `ambush()` rolls the mutually-exclusive first-strike flags (preemptive beats back attack) — the game reads them to pre-fill the party's or the enemies' ATB gauges via `battle.setRate`/gauge lead. `escapeChance({ partySpeed=1, enemySpeed=1, attempts=0 }) -> 0..1` powers the Run command: it rises with the party's speed share of the room and by a flat bump per FAILED attempt (a cornered party eventually breaks free), clamped. Zone tuning: `setRate(r)` swaps the base chance (calm town vs deep dungeon), `danger(m)` scales it (repel/curse items, a thinned bestiary) — both chainable; `reset(g=grace)` is called ONCE after a fight to clear the meter and arm the safe window; `threat()`/`grace()`/`rate()` introspect for a HUD. Pure, deterministic, node-safe (no window/DOM/timers/Math.random; all state — meter, grace, rate, danger — in the closure), sim-tested: higher rate yields more encounters over N steps, grace suppresses an instant re-encounter, the weighted pick honours weights, and escape rises with attempts.

## Post-battle rewards / loot (reusable component — engine/control/rewards.js)
The SPOILS half of an FF6-style fight — the turn-based sibling of the field-layer economy/inventory/progression systems, run when battle.js ends in victory to turn fallen enemy defs into XP / gil / AP + item drops, split the XP across the party, and push anyone past a level threshold through the round-9 progression/stats stores. PURE DATA + INJECTED rng (a `() => 0..1` fn — a deterministic `() => 0.5` in tests — or a core makeRng `{next}`); no engine imports, no window/DOM/timers/Math.random, so a battle scene, a results screen and a headless balance sim get the SAME numbers. An ENEMY def is plain data `{ xp, gil (or gold), ap, drops:[{item,chance}], steal:[{item,chance}] }`; a PARTY MEMBER just needs an `id` and SOME way to hold a level — a `levels` (progression.makeLevels) store, a `stats` (stats.js) block, or a bare `{ level }`. `rewards.computeRewards(defeatedEnemies, { rng, bonus? }) -> { xp, gil, ap, drops:[items] }` SUMS xp/gil/ap across the list and rolls each enemy's drop table INDEPENDENTLY (each slot on its own `chance`), optionally scaling the totals by a victory `bonus`. `rewards.distribute(xp, party, { rng?, split=true, survivorsOnly=true, all?, curve? }) -> [{ id, gained, from, to, leveledUp, levels }]` splits the pool across SURVIVORS by default (FF6: KO'd members earn nothing; `all:true`/`survivorsOnly:false` lets the fallen share, `split:false` gives everyone the full amount), banks each share through the member's growth store (or accumulates on `member.xp` against an injected `curve(level)->xpNeeded`), and on a level-up re-derives the stats block via `stats.setLevel` so maxHp climbs — returning WHO leveled. `rewards.lootTable(entries) -> { entries, rollAll(rng) (independent per-slot rolls — the drop path), rollOne(rng) (weighted single pick by `chance` — the steal path) }` is the shared roller. `rewards.steal(target, rng, { rate=0.5 }) -> { success, item }` is the Thief command: a rate gate then a weighted pick from the target's `steal` table (success can be true with a null item when there's nothing left). `rewards.bonusFor(conditions, table?) -> { xp, gil, ap, tags }` folds active named victory conditions (defaults `noDeath` 1.5× xp/ap · `flawless` · `overkill` 2× gil · `quick` · `preemptive` · `noItems`) into STACKING multipliers, and `rewards.applyBonus(rewards, bonus)` scales a spoils bag by a bonusFor result OR a raw conditions bag. Deterministic and node-safe; sim-tested by asserting drops honor chance under a seeded rng, XP splits and triggers a level-up at the threshold, steal respects rate, and gil/ap sum.

## Battle commands (reusable component — engine/control/commands.js)
The FF6 signature that every character owns a UNIQUE battle menu — the verb layer atop the ATB core (battle.js owns timing, formulas.js/spells.js own the numbers, this owns "what can this unit do and what does picking it do"). A command is data + one resolver: `{ name, kind:"fight"|"magic"|"item"|"skill", target?, mp?, enabled?(user), resolve(user, targets, ctx) -> results[] }`. `commands.makeCommandSet(list)` -> `{ list(user) (visible+enabled verbs — a status-sealed command drops out), get(name), has(name), all(), run(name, user, targets, ctx) }`. `run` is the gate: it refuses unknown/disabled commands, blocks an MP-gated one when the user is short (spending nothing), else resolves and spends MP — delegating the actual EFFECT to INJECTED resolvers on ctx (`ctx.formulas`/`ctx.spells`/`ctx.rng`/`ctx.spellbook`/`ctx.inventory`), so this file hard-imports nothing and stays node-testable on plain objects. Baseline verbs every hero shares: `attack()` (Fight — physical via ctx.formulas), `magic()` (opens the spellbook — ctx.spell cast via ctx.spells), `item()` (use a consumable). FF6-style uniques as data-driven examples: `steal()` (rng grab from target.steal) · `blitz({input})` (fighting-game combo — fires only on the matching directional input) · `swdTech({tiers})`/`charge` (chargeable tier attack, top tiers sweep a side) · `tools()` (defense-ignoring machines with status riders) · `lore()` (learned enemy skills) · `hurl()`/throw (one-shot thrown weapon). `commands.baseCommands(extra)` spreads a Fight/Magic/Item menu. Pure, deterministic (inject rng), node-safe.

## Calendar / game time (reusable component — engine/control/calendar.js)
GAME TIME + calendar — the Stardew-style clock the whole farm sim runs on. `calendar.makeCalendar({ dayStartHour=6, dayEndHour=26, minutesPerTick=10, seasonDays=28, seasons=["spring","summer","fall","winter"], startDay=1, duskHour=18, startYear=1 })` returns a stateful clock you feed game-minutes. `advance(minutes) -> events[]` rolls the wall clock and returns the discrete happenings crossed, each `{ type }`: `tick {hour,minute}` (every `minutesPerTick`) · `hour {hour}` (on the :00) · `dayRolled {day,season,seasonIndex,year,dayOfWeek}` (at midnight) · `seasonRolled {season,seasonIndex,year}` (day 28 -> next season) · `yearRolled {year}` (season 4 -> next year). A play-day runs from `dayStartHour` (6:00) up to `dayEndHour` (26:00 = 2am), where the farmer PASSES OUT: `api.passOut` latches true and `advance()` freezes until you `sleep()`. `sleep() -> events[]` fast-forwards to the next `dayStartHour` morning, emitting ONLY the roll events (dayRolled/seasonRolled/yearRolled) and clearing passOut. Crossing midnight (24:00) already flips the DATE to the next day before you sleep (Stardew), so there is never a double day-roll. Readers: `clock() -> {hour,minute}` (24h wall clock) · `date() -> {day (1..seasonDays), season, seasonIndex, year, dayOfWeek}` · `isNight` (getter: dark before morning or past dusk) · `format() -> "Mon, Spring 1, 6:00"`. Pure, deterministic, node-safe (no Date/now/timers/Math.random/DOM); all state in the closure and fractional minutes carry losslessly so real-time `dt` feeds it cleanly. Compose the rest of the farm sim (crops, energy, saves, quests) by subscribing to the plain-data event stream rather than importing this — e.g. `for (const ev of cal.advance(dt*60)) if (ev.type==='dayRolled') growAllCrops()`.

## Crops / farming (reusable component — engine/control/crops.js)
The Stardew-style daily plant → water → grow → harvest loop as a PURE, headless soil sim — no imports, no globals, no Math.random, so it runs deterministically under node and composes with the world/economy/inventory/save layers via PLAIN DATA. A crop DEF is a data-table row: `{ id, name, seasons:['spring'], stages:[days per stage], produce, regrow?, yield?, seedPrice?, sellPrice? }` (total grow days = sum of `stages`; `regrow` = days to ripen again after each harvest; `yield` = units/harvest, default 1; a crop with no `seasons` grows year-round). A PLOT is per-farm-tile soil: `{ tilled, crop, stage, dayInStage, watered, dead, readyToHarvest, regrowLeft }` — make one per tile (index by `tilemap.worldToTile`), never share refs. API: `makeCropbook(defs) -> { get(id), all(), inSeason(id, season) }` (indexes a crop table, never mutates it) · `makePlot() -> plot` · `till(plot)` (hoe soil, idempotent) · `plant(plot, cropDef, season) -> bool` (fails on untilled/occupied/off-season) · `water(plot)` (set today's watered flag on tilled soil) · `growDay(plot, { season, rained }) -> plot` (advance ONE day: a live crop out of its `seasons` withers overnight; an unwatered + unrained plot stalls with no progress lost; a watered-or-rained crop walks its stages and flips `readyToHarvest` after the summed stage-days; a ripe crop sits waiting) · `harvest(plot) -> { produce, qty } | null` (reaps a ripe plot; a `regrow` crop resets to its `regrowLeft` timer and keeps its roots to produce again, a one-shot crop is pulled leaving tilled soil to replant) · `endOfDayReset(plot)` (evaporate the water flag each morning, before re-watering). Typical day: player tills/plants/waters in the morning; at nightfall the game calls `growDay(plot, { season, rained })` then `endOfDayReset(plot)` for every plot. Bundle `crops = { makePlot, makeCropbook, till, plant, water, growDay, harvest, endOfDayReset }`. Sim-tested: watered+growDay advances, unwatered stalls, matures after summed stage days, regrow keeps producing, off-season plant blocked and a live crop dies at season change.

## Weather (reusable component — engine/control/weather.js)
DAILY WEATHER oracle for a farm/foraging sim — rolled each morning to shape the day (rain auto-waters the farm, storms crackle with lightning, snow blankets winter). `makeWeather({ rng, tables })` returns `{ roll, today, forecast, effects }`; it holds NO world state, only the last committed roll and one peeked-but-uncommitted forecast in its closure, so it's pure and node-safe. `rng` is the injected 0..1 source — a bare `()=>0..1` fn (test default `()=>0.5`), a core `makeRng` object (`{next()}`), or omitted (deterministic 0.5). `tables` maps a season to a weighted list `[{ weight, kind }]` over kinds `sun|rain|storm|wind|snow`, merged over sensible defaults (`DEFAULT_TABLES`): spring is rainy, summer bakes with rare storms, fall is windy, winter snows and NEVER rains (no rain/storm entries). API: `roll(season) -> { kind, season, ...flags }` advances to a new day, consuming a matching `forecast` so the TV never lies, else drawing fresh, and updates `today()`; `today()` returns the last committed roll (null before the first); `forecast(season)` peeks the NEXT day without committing (cached — a repeated peek is stable and the following `roll` returns exactly it). `effects(kind)` returns a FRESH copy of the descriptive flag set a game reads: `rainsCrops` (rain/storm skip the watering can), `lightning` (storm), `indoorsBonus` (wet/cold days nudge you inside — crafting/social), `fishingBonus` (rain/storm), `foragingBonus` (wind), `wateringNeeded` (dry days). Deterministic given the rng — a seed replays the exact same season of weather. Compose by PLAIN DATA: read the booleans off the roll and drive your own farm/fishing/energy systems (`if (day.rainsCrops) farm.waterAll()`). Bundle `weather = { makeWeather, effects, KINDS, EFFECTS, DEFAULT_TABLES }`.

## Tools (reusable component — engine/control/tools.js)
The verbs a farmer SWINGS at a tile or object — hoe/wateringCan/axe/pickaxe/scythe/fishingRod — sitting between input and the soil sim (crops.js) / world objects. A swing costs energy (energy.js vitals) then mutates what it hits; pure DATA + injected helpers (a vitals, an rng, a plot lookup), so it runs headless. A tool DEF is `{ name, action:"till"|"water"|"chop"|"mine"|"cut"|"fish", energy?, tier?, area?, capacity? }`; build with `tools.makeTool(def)` or a named factory (`tools.hoe()`, `tools.wateringCan()`, `tools.axe()`, `tools.pickaxe()`, `tools.scythe()`, `tools.fishingRod()`). `tier` (basic->copper->steel->gold->iridium) widens the AoE for till/water and lowers energy per swing; `tools.upgrade(tool, tier)` returns a fresh tool carrying a can's water over. `tools.useTool(tool, target, ctx) -> { ok, effect }`: energy GATES the swing (an exhausted/broke farmer gets `ok:false`, nothing mutated, no energy spent); otherwise it spends via `ctx.vitals.spend`, applies the verb, and returns `effect = { action, drops:[{item,qty}], ...flags }`. Verbs: till (soil.tilled), water (tilled plot.watered, drains the can which can run dry mid-swing), chop (tree hp over hits -> wood), mine (rock/ore hp over hits -> stone/ore), cut (scythe clears grass/forage, hay on an rng roll), fish (rng bite vs catchChance, then pick from the spot/tool fish table). For an area tool pass `ctx.center=[tx,tz]` + `ctx.plotAt(tx,tz)` and it applies to every tile in `tools.areaTiles(tool, center)`. `tools.energyCost(tool)` previews the swing cost. Deterministic (inject rng), node-safe; drops are plain `{item,qty}` for inventory/economy.

## Relationships (reusable component — engine/control/relationships.js)
NPC FRIENDSHIP + gifting — the social spine of a Stardew-like life-sim. A game holds ONE `relationships = relationships.makeRelationships({ maxHearts=10, pointsPerHeart=250, giftsPerWeek=2, decay=2 })` and tracks a point-based bond per NPC (lazily created on first reference). Points roll up into HEARTS (`points / pointsPerHeart`, floored, capped at `maxHearts`) — the number the social UI shows and other systems gate on. API: `meet(id)` (register, idempotent) · `known(id)` · `points(id)` · `hearts(id)` · `add(id, n)` (raw friendship for quests/events/cutscenes; clamped to [0..cap], positive spares today's decay) · `gift(id, item, prefs) -> { delta, reaction }` where `reaction` is `"love"|"like"|"neutral"|"dislike"|"hate"` from the NPC's taste table `prefs = { loved:[], liked:[], disliked:[], hated:[] }` (love +80 · like +45 · neutral +20 · dislike -20 · hate -40), rationed to `giftsPerWeek` — an over-cap gift lands nothing and returns `reaction:"none"` · `talk(id) -> { delta, talked }` (a daily hello, `talkBump` points once/day, later chats no-op) · `decayDay(exceptIds)` (each morning: every KNOWN bond you didn't touch today loses `decay`, then daily flags reset; pass an array/Set of pinned NPCs — spouse/best friends — to spare) · `dayReset()` (clear just the once-a-day talk lock) · `weekReset()` (refill everyone's gift allowance on the week rollover) · `status(id) -> { hearts, points, giftedThisWeek, talkedToday }` · `toJSON()`/`load(data)` for save.js. Pure DATA + closures, fully deterministic (no rng/DOM/timers/Math.random); compose with calendar.js (call `decayDay` each morning, `weekReset` on week-start), dialogue.js, quests.js via plain calls — never hard imports.

## NPC schedules (reusable component — engine/control/schedules.js)\nWHERE each villager is at each minute — the lived-in-town brain, tied to calendar.js. A schedule is DATA: `{ npcId, entries:[{ at:"HH:MM"|minutes, place, pos:[x,z], action? }], variants?:[{ dayOfWeek?, season?, weather?, hearts?, entries }] }`. `schedules.makeSchedule(def)` -> `{ npcId, entries, at(timeMinutes, ctx?), place(t,ctx?), posAt(t,ctx?), next(t,ctx?), dayFor(ctx) }`: `at()` returns the active block (the latest entry whose start time has passed; before the first it WRAPS to the last, i.e. the overnight spot), `next()` the upcoming block (wraps to tomorrow's first), `posAt`/`place` the conveniences. Time is MINUTES-OF-DAY (the `hour*60+minute` calendar.js reports). `schedules.makeDirector([...schedules])` -> `{ update(time, ctx) -> [{ npcId, entry, moved }], positions(time, ctx), add(s), reset() }`: `update` surfaces ONLY the NPCs whose active block just CHANGED (moved:true) so a game paths just those to `entry.pos`; the first call reports everyone (initial placement), `positions()` is a full snapshot that ignores change-tracking, `reset()` forces a re-emit. Variant selection scores each variant's guard by specificity (every declared key must match; `hearts` is a friendship MINIMUM; unspecified keys are wildcards) and picks the TIGHTEST match, else the default day — so a rainy Saturday beats plain rain beats the default. ctx = `{ dayOfWeek, season, weather, hearts }` where `hearts` is a number OR a `{ npcId: n }` map (one director serves the whole town). Pure, deterministic, node-safe (no Date/DOM/timers/Math.random; state in closures), sim-tested by asserting the active block at a given minute, that a new block at its start time triggers a director move, and that a rain/day variant overrides the default.

## Skills (reusable component — engine/control/skills.js)
The Stardew life-sim SKILL layer — five parallel XP tracks (farming/mining/foraging/fishing/combat, or your own), each climbing its OWN curve to a level cap, unlocking perks and branching professions. A sibling of progression.js but shaped for a farm sim: many tracks + auto perks + exclusive career picks. Pure DATA (xp/level/picks in a closure — no Math.random, no timers), so a HUD bar, a level-up popup, a perk screen and a balance sim drive the SAME store. `skills.makeSkills({ skills=["farming","mining","foraging","fishing","combat"], curve=stardewCurve(), maxLevel=10 })` -> `{ xp(skill), level(skill), addXp(skill,n)->{level,leveledUp,gained} (rolls multiple level-ups, caps at maxLevel), progress(skill) (0..1 to next), xpToNext(skill), totalLevel() (sum, gates events/marriage), definePerks(skill,[{level,id,choice?,requires?}]) (chainable), choices(skill,level), choose(skill,level,perkId)->bool (exclusive per tier; fails until the level is reached and any requires-gate is met), earned() (auto perks whose level is reached), chosen() (professions actively picked), has(perkId) (the gameplay query: auto-earned OR chosen), list(), toJSON()/load() for save.js }`. A perk def with no `choice` is an AUTO perk (on the moment level>=its level); `choice:true` marks a mutually-exclusive PROFESSION at its tier (pick one via choose, re-picking replaces it); `requires:'perkId'` gates a later pick on an earlier one (Stardew's level-5 -> level-10 branch). A `curve(level) -> xp to climb FROM level to level+1` sets the pace; default `stardewCurve()` matches the real per-skill table (incremental 100/280/390/530/850/1150/1500/2100/3100/5000 = cumulative 100..15000), plus `linearSkillCurve(base)`. `skills` bundle = `{ makeSkills, stardewCurve, linearSkillCurve }`. Node-safe and sim-testable (add xp to one skill and assert isolated multi-level climb, an auto perk flips at its level, a profession is exclusive per tier, has() reflects earned+chosen).

## Fishing minigame (reusable component — engine/control/fishing.js)
The Stardew CATCH BAR as a pure, deterministic real-time STATE MACHINE a UI drives — the fishing counterpart to the round-11 farm core, composed by plain data (tools.js decides a bite, weather.js `fishingBonus` can ease difficulty, then this owns the tug-of-war and the caller hands `caught` fish to inventory). `fishing.makeFishingBar({ fishSpeed=0.5, barSize=0.3, fillRate=0.4, drainRate=0.35, difficulty=0.4, seed=1, startProgress=0.35, catchSpeed=0.55, escapeSpeed=0.4, bounce=0.3, force=24, treasure=false })` models a vertical 0..1 bar: a seeded fish bobs toward reselected targets (behavior flavors `smooth·mixed·dart·sinker·floater`, harder fish retarget faster and dart farther) via frame-rate-independent exponential approach, while the GREEN bar accelerates UP while the player HOLDS (reeling) and falls by gravity when released, rebounding off the walls. `step(dt, holding) -> { fishPos, barPos, barTop, barBottom, barSize, progress (0..1), overlapping, done, caught }`: progress climbs (`catchSpeed`) while the fish is inside `[barBottom,barTop]` and bleeds (`escapeSpeed`×difficulty) while it escapes; CATCH at `progress>=1`, FAIL (line snaps) at `progress<=0`, after which the machine freezes idempotently. `cast(fish)` hooks a NEW fish and resets the run from plain data `{ difficulty?, behavior?, treasure? }`; `get()` peeks the snapshot; `reset(fish)` re-arms. The `treasure` flag adds a STATIONARY second target with its own `treasurePos/treasureProgress/treasureCaught/overlappingTreasure` meter that must fill BEFORE the fish is caught (freeze-on-catch loses an unfilled chest, matching Stardew). All randomness is an injected integer `seed` through an internal mulberry32 rng (no Math.random), all state lives in the closure — node-safe and sim-testable by stepping with `holding=true/false`, asserting progress rises on overlap, falls off it, and a run of good holds lands the catch.

FARM ANIMALS — the coop/barn husbandry loop (Stardew: chickens/cows/sheep/…), the livestock sibling of crops.js: animals turn HAY + AFFECTION into produce overnight. Pure DATA + closures, no world/DOM/clock/Math.random — you hand `dayPass` the day's facts and it advances one animal one night, deterministically. An ANIMAL is plain data `{ id, kind, name, age (days owned), happiness (0..maxHappiness), fed (TODAY), pettedToday, produce ({item,qty,quality,needsTool?}|null), produceReady, recipe }`. Build one with `animals.makeAnimal({ id, kind, name?, age?, happiness?, produceIn? })`, which stamps the kind's recipe from `animals.KINDS` (chicken/duck/rabbit/cow/goat/sheep/pig — each `{ item, qty, matures (days to first produce), interval (days between), needsTool? }`; def.recipe overrides win). A BARN/coop `animals.makeBarn({ capacity=4 }) -> { capacity, add(a) (false if full/dup-id), remove(id)->animal|null, get(id), list() (shallow copy), count, isFull() }`. Daily loop: `animals.feed(animal)` sets the day's ration (idempotent per day, cleared each dayPass); `animals.feedAll(barn, hay) -> { fed, hayLeft }` rations the whole barn from a HAY COUNT until it runs out; `animals.pet(animal, { bump=5 })` is a once-a-day happiness bump (false if already petted). `animals.dayPass(animal, { fed, housed=true, weather }) -> { produced:{item,qty,quality,needsTool?}|null, mood }` advances one night: `fed` defaults to `animal.fed`; adjusts happiness (fed +2, hunger −20, storm/snow exposure when housed=false −20, un-petted neglect −4); ages the animal; and — only if fed, sheltered, and grown (produceIn counts matures→interval) — makes produce for the morning (unfed or storm-exposed animals lose happiness and skip produce); then clears the fed/pettedToday flags. `animals.collect(animal, { tools? }) -> the ready produce (clearing it) or null`; when the produce `needsTool` (pail/shears) and a `tools` set/array/flag-map is passed that lacks it, collection fails and the produce stays put (no `tools` arg = permissive). Produce QUALITY (regular|silver|gold) rises monotonically with happiness plus a small maturity nudge (age ≥ 40). Weather may be a weather.js string ('storm'/'snow'/'blizzard'/'thunder' are bad) or an object flag `{ bad|storm|snow }`. Deterministic and node-safe; drops are plain `{item,qty,quality}` for inventory/economy. Bundle: `export const animals = { makeBarn, makeAnimal, feed, feedAll, pet, dayPass, collect, KINDS }`.

## Building a game from a description (the app's core job)
The app's purpose is making games. Given a description, DON'T write an engine — pick components and glue:
1. **View + movement** from the description: top-down shooter → `cameras.topDown` + `movements.twinStick`; platformer → `cameras.sideScroller` + `movements.platformer`; runner → `cameras.sideScroller` + `movements.autoRun`; true 2D → `cameras.flat2D`.
2. **Systems** in `game/bootstrap.js`: keep the ones the game needs (movement, ai, collision, health, score, hud, audio, render, gamestate are almost always in; add `fire`/`spawn` for shooters, drop them for a pure platformer). For a shooter also pick `ctx.weapon` (single/rapid/shotgun/burst/radial) and `ctx.aim` (stick/facing/manual/autoAim) — e.g. a one-thumb mobile shooter is `aim.autoAim` + `weapons.rapid`. Pick enemy/NPC brains with `ctx.behavior` (chase/orbit/zigzag/charger for foes; wander/patrol/follow/guard for NPCs), or `behaviors.byKind({...})` to mix.
3. **Entities** (`game/entities.js`): mesh (box/sphere/cylinder/plane), color, size, stats — pure data. **Config** (`game/config.js`): the numbers.
4. **Signals/registries** (`game/contracts.js`): add any the game needs.
5. Ask the user AT MOST 1–2 questions only for genuinely ambiguous core choices (e.g. "waves or endless? health or one-hit?"). Otherwise pick sensible defaults and build.
6. Only write a NEW system for behavior no component provides — one small file, imports only core, reacts to events/signals.

You should not need to open `engine/core`, `engine/render`, `engine/control`, or the existing `engine/systems`.
